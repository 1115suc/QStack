
### 1. 生产环境出现了问题，你们是如何排查的?

在微服务架构下，生产环境的问题通常分为**异常错误**和**性能问题**两类。由于服务实例众多且分散，我们不再使用传统的逐台服务器登录查看日志的方式，而是采用**分布式日志系统**配合**链路追踪系统**进行排查：

*   **针对异常报错**：
    *   我们使用 **Graylog** 进行集中式日志管理。
    *   当出现问题时，我会登录 Graylog 控制台，通过全局唯一的业务 ID（如订单号）或报错关键字进行搜索。
    *   Graylog 支持高亮显示和快速过滤，能迅速定位到是哪个微服务节点抛出的异常，并查看完整的堆栈信息。
*   **针对链路/性能问题**：
    *   如果问题表现为“请求变慢”或“流程中断”，单看日志很难串联上下文。
    *   我们会结合 **Skywalking** 链路追踪工具。通过 TraceID 查找整个请求的调用链，查看在哪个环节耗时最长，或者在哪个服务节点调用失败。

### 2. 你们的日志是如何存储的?

我们的日志存储方案采用的是 **Graylog + Elasticsearch** 的架构，具体流程如下：

*   **收集端**：
    *   微服务项目中集成了 `logstash-gelf` 依赖，配置 Logback 使用 **GELF UDP Appender**。
    *   日志直接从应用端以 UDP 协议推送到 Graylog Server，减少对业务性能的影响。
*   **存储端**：
    *   Graylog Server 接收到日志后进行格式化处理，最终将日志数据存储在 **Elasticsearch** 集群中，利用 ES 强大的倒排索引能力提供快速搜索。
    *   Graylog 的配置信息（如用户、输入源配置）则存储在 **MongoDB** 中。
*   **回收策略**：
    *   为了防止海量日志占满磁盘，我们在 Graylog 中配置了**日志回收策略**（Rotation Strategy）。
    *   通常根据日志数量（Index Message Count）或索引大小（Index Size）进行限制，自动清理过期的历史日志索引。

### 3. web接口性能比较差，你会如何排查?

接口性能差通常意味着调用链中存在“慢节点”。我会使用 **Skywalking** 进行 APM（应用性能监控）排查：

1.  **定位慢链路**：在 Skywalking 的追踪（Trace）页面，找到响应时间过长的请求记录。
2.  **分析 Span**：点击查看该链路的详细 **Span** 信息。Skywalking 会以瀑布流的形式展示每一个环节（数据库调用、RPC 调用、缓存读取等）的耗时。
3.  **确定瓶颈**：
    *   如果是 **Database Span** 耗时很长，说明是 SQL 需要优化或索引失效。
    *   如果是 **RPC Span** 耗时很长，说明下游微服务处理慢或网络存在延迟。
    *   通过这种可视化的方式，可以精准定位是哪一行代码或哪一次外部调用拖慢了整体接口。

### 4. 微服务间的调用关系比较复杂，你会如何梳理?

在几十甚至上百个微服务的系统中，人工梳理代码逻辑非常困难。我们主要依赖 **Skywalking 的拓扑图（Topology）** 功能：

*   **自动探测**：Skywalking 的 Agent 会在服务运行时自动收集服务与服务之间、服务与数据库/中间件之间的调用数据。
*   **可视化展示**：
    *   在 Skywalking UI 的拓扑图界面，能清晰地看到整个系统的架构图。
    *   它展示了节点（微服务、DB、Redis）以及节点之间的连线（调用关系）。
    *   如果连线是红色的，还能直观地发现当前服务间调用存在告警或错误。
*   这不仅帮助我们梳理业务依赖，在新员工入职熟悉架构或进行系统重构时也提供了极大的帮助。

### 5. 你们项目中使用什么性能监控工具?

我们项目选型的是 **Apache Skywalking** 作为 APM 工具。

**选择它的原因主要有以下几点：**

1.  **高性能探针**：相比于 Zipkin，Skywalking 的 Agent 对微服务吞吐量的影响更小，性能更高，适合生产环境。
2.  **无侵入性**：它使用 Java Agent 技术（启动参数配置 `-javaagent`），**对代码完全透明**。我们不需要修改任何业务代码就能实现监控，方便启用和禁用。
3.  **功能完善**：它不仅支持链路追踪，还提供了完整的**服务性能指标监控**（如响应时间、吞吐量、SLA）以及**自动化的应用拓扑图**。
4.  **扩展性强**：后端 OAP Server 支持集群部署，存储支持 Elasticsearch，能够支撑大规模分布式系统的监控数据。
