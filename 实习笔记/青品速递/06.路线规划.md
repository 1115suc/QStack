
# 面试题回答：路线规划微服务

## 1. 你们物流项目中的路线规划是怎么做的？

**回答：**

在我们的物流项目中，路线规划是核心功能之一，主要负责计算从发件网点到收件网点之间最优的运输路径。考虑到路线数据是一种典型的网状结构，且需要进行深度的路径查询（例如查找两点间的最短路径），我们没有使用传统的关系型数据库（MySQL），而是选用了 **图数据库 Neo4j** 来实现。

具体实现流程如下：

1. **数据模型构建**：
    
    - 我们将物流网络中的机构（一级转运中心 OLT、二级分拣中心 TLT、网点 Agency）映射为图数据库中的**节点（Node）**。
        
    - 将机构之间的运输路线（干线、支线、接驳路线）映射为**关系（Relationship）**。
        
2. **路线查询**：
    
    - 利用 Spring Data Neo4j 框架与数据库交互。
        
    - 使用 **Cypher 查询语言**编写核心查询逻辑。
        
    - 我们实现了两种核心的规划策略：
        
        - **最短路径策略**：使用 Neo4j 内置的 `shortestPath` 函数，查询节点间跳数最少的路径，适用于时效优先场景。
            
        - **成本最低/权值优先策略**：通过查询所有可行路径，并累加路径上关系（Relationship）的 `cost` 属性，按成本和路径长度排序，取最优解。
            
3. **微服务架构**：该功能封装在 `sl-express-ms-transport` 微服务中，对外提供 Feign 接口供订单系统或调度系统调用。
    

## 2. 如何确定路线的成本和距离？成本计算规则是什么？该成本会计算到公司利润核算中吗？

**回答：**

关于路线属性的确定和计算，我们有一套完整的逻辑：

- **距离确定**：
    
    - 距离不是随意填写的，而是基于真实的地图数据。我们使用了自研的地图中台服务 **EagleMap**（底层对接了高德地图/百度地图 API）。
        
    - 在新增或编辑路线时，系统会根据起点机构和终点机构的经纬度，调用 EagleMap 的路径规划接口（Driving 模式），获取真实的**行驶距离（米）**和**预计耗时（秒）**。
        
- **成本计算规则**：
    
    - 成本是自动计算的，公式为：`路线成本 = 距离(km) * 每公里平均成本`。
        
    - 其中“每公里平均成本”是根据路线类型（干线、支线、接驳）配置在 Redis 中的（默认配置如干线 0.8元/km，支线 1.2元/km）。
        
    - 代码逻辑中，我们先通过地图获取距离，再读取 Redis 中的配置系数，最后得出该条路线的预计成本并存储到 Neo4j 的关系属性中。
        
- **关于利润核算**：
    
    - **不会**。这里的成本仅用于**路线规划时的算法权重参考**（即用于选出“理论上”成本最低的路线）。
        
    - 真实的运输成本极其复杂（包含油费、过路费、司机工资、车辆折旧等实时变动成本），不能仅凭这个理论计算值作为财务核算依据。文档中也明确备注了，该成本“不能作为真实的成本进行利润计算”。
        

## 3. 对于路线的往返你们是怎么设计的？为什么成对创建的？

**回答：**

在我们的设计中，路线必须是**双向往返**的。

- **设计逻辑**：
    
    - 现实生活中，车辆从 A 点运输到 B 点，通常也需要从 B 点运回 A 点，或者至少路网是双向联通的。
        
    - 在 Neo4j 的图结构中，关系（Relationship）是有方向的（Directed）。如果我们只创建了 `A -> B` 的关系，那么在计算从 B 到 A 的路径时，图算法将无法通行。
        
    - 因此，为了保证全网的连通性，必须显式地维护双向关系。
        
- **实现方式**：
    
    - 在代码层面（Repository层），我们在创建路线时使用了 Cypher 语句一次性创建两条关系。
        
    - 具体逻辑是：同时创建 `(A) -[r1]-> (B)` 和 `(B) -[r2]-> (A)`。
        
    - 这两条关系共享相同的物理属性（如距离、预估时间），但代表了不同的流向，确保了路线规划算法在任何方向上都能找到通路。
        

## 4. 路线支持修改起点或终点机构吗？请说明理由。

**回答：**

**不支持**直接修改起点或终点机构。

- **技术理由**：
    
    - 在图数据库（Neo4j）中，**关系（边）是强依赖于起点节点和终点节点的**。一条边唯一地连接两个特定的节点。
        
    - 如果修改了起点或终点，从图论的角度来看，这实际上已经不再是原来的“那条边”，而是一条全新的边（连接了不同的节点）。
        
    - 在我们的代码实现中（`TransportLineServiceImpl.updateLine`），在进行属性拷贝时，明确使用了 `IgnoreProperties` 忽略了 `startOrganId` 和 `endOrganId` 字段，禁止更新这两个属性。
        
- **业务理由**：
    
    - 修改起止点等同于废弃旧路线并开通新路线。
        
    - 正确的操作流程应该是：**删除**旧的路线，然后**新增**一条新的路线。这样也能保证历史数据的完整性和图结构的清晰。
