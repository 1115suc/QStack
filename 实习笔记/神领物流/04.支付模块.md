# 支付模块

### 1. 来，聊聊分布式锁的原理？自己实现分布式锁的难点在哪里？

**原理：**
分布式锁的核心思想是**互斥**。在分布式系统中，多个微服务节点（JVM）无法共享本地锁（如`synchronized`或`ReentrantLock`），因此需要借助一个外部的、所有节点都能访问的中间件（如Redis、Zookeeper）来实现锁。
基于Redis的实现原理主要是利用 `SETNX` (SET if Not eXists) 命令。
*   **加锁：** 执行 `SETNX key value`，如果返回1（成功），说明获取锁成功；如果返回0，说明锁被占用。
*   **解锁：** 删除该 `key`。

**自己实现的难点（踩坑点）：**
1.  **原子性问题：** 加锁和设置过期时间必须是原子的。如果`SETNX`成功但没来得及设置过期时间机器宕机，会导致死锁。需要使用 `SET key value EX seconds NX` 复合命令或Lua脚本。
2.  **误删锁问题：** 线程A获取锁，执行时间超过了过期时间，锁自动释放；线程B获取锁；此时线程A执行完毕去删除锁，把线程B的锁删掉了。
    *   *解决：* `value`必须存入唯一标识（如UUID+线程ID），删除前判断`value`是否属于当前线程。判断和删除也需要原子性（Lua脚本）。
3.  **锁续期问题（看门狗）：** 很难预估业务执行时间。如果设置5秒过期，业务执行了6秒，锁会提前失效，导致并发安全问题。需要一个后台线程（看门狗）定期检查并给锁“续命”。
4.  **可重入性：** 简单的Redis锁不可重入，需要使用Hash结构存储锁的持有次数。

### 2. 使用Redisson实现分布式锁具体是怎么使用的？聊聊看门狗是个啥，它有什么用？

**具体使用：**
在项目中（参考文档 `day02` 第6.5节），通过引入Redisson依赖并配置`RedissonClient`：
1.  注入 `RedissonClient`。
2.  获取锁对象：`RLock lock = redissonClient.getLock("lock:order:" + orderId);`（支持公平锁/非公平锁）。
3.  加锁：`lock.tryLock(waitTime, leaseTime, unit)` 或者直接 `lock.lock()`。
4.  业务执行（放在 `try` 块中）。
5.  解锁：`lock.unlock()`（放在 `finally` 块中，且建议判断 `lock.isLocked() && lock.isHeldByCurrentThread()`）。

**看门狗（Watchdog）机制：**
*   **是个啥：** 它是Redisson内部的一个后台定时任务（TimeTask）。
*   **有什么用：** 解决**锁过期时间短于业务执行时间**的问题。
*   **工作原理：**
    *   如果在加锁时**没有指定** `leaseTime`（锁持有时间），Redisson会默认设置锁的过期时间为30秒（`lockWatchdogTimeout`）。
    *   加锁成功后，后台会启动一个看门狗线程，每隔 `lockWatchdogTimeout / 3`（默认10秒）检查一次。
    *   如果当前线程仍然持有锁，看门狗会将锁的过期时间重新重置为30秒。
    *   这样可以保证只要业务线程还在运行且未释放锁，锁就不会因为超时而自动释放。当业务完成调用 `unlock()` 或服务宕机（看门狗线程随之挂掉）时，锁才会被释放。

### 3. 支付二维码的生成你们是在服务端还是在客户端？为什么不在服务端？在客户端生成有什么好处？

**生成位置：**
我们在**客户端（前端/小程序端）**生成二维码，服务端只负责调用支付接口获取支付链接（如微信的 `code_url` 或支付宝的 `qr_code` 字符串），并返回给前端。

**为什么不在服务端 & 好处：**
1.  **节省服务器资源：** 生成二维码图片需要进行图形渲染，比较消耗CPU和内存资源。将此压力转移到客户端，能显著降低服务端负载。
2.  **节省带宽：** 传输一个字符串链接（几十字节）比传输一张图片（几KB到几百KB）要快得多，节省网络带宽，响应速度更快。
3.  **灵活性：** 前端拿到链接后，可以利用前端组件库（如 `qrcode.js`）灵活控制二维码的大小、颜色、样式，无需服务端频繁调整。
    *(参考 `day03` 第4.6节 优化练习部分)*

### 4. 用户在选择支付渠道后，可能会反复的切换支付渠道，这块你们是怎么处理的？

这块涉及到**幂等性**和**交易单更新**逻辑（参考 `day03` 第4.3节 `idempotentCreateTrading` 方法）：

1.  **查询现有交易单：** 用户发起支付时，先根据订单号查询数据库中是否已存在交易单。
2.  **状态判断：**
    *   如果状态是“已支付”，直接抛出异常或返回成功，防止重复支付。
    *   如果状态是“付款中”（FKZ）：
        *   **渠道相同：** 如果之前选的是支付宝，现在还是支付宝，则返回原有的支付链接/二维码（幂等性）。
        *   **渠道不同：** 如果之前是支付宝，现在切换成微信。此时不能修改原交易单的支付渠道字段（因为原渠道的订单在第三方平台可能已存在）。
        *   **处理方式：** 我们会**保留原有的交易单记录**，将其ID赋给当前对象，但**重新生成一个全新的交易单号（TradingOrderNo）**，然后调用微信接口下单，并更新数据库中的交易单号和渠道。
    *   **核心点：** 保证一个业务订单（ProductOrderNo）对应一个交易单记录，但通过更新“交易流水号”来适配不同的支付渠道请求，避免在第三方支付侧产生冲突。

### 5. 用户在申请支付后，发现并没有展示出二维码，这时用户可能会不断的重试，你们有考虑这种情况吗？具体是怎么处理的？

**考虑了，主要通过分布式锁和幂等性校验来处理：**

1.  **分布式锁（防并发）：**
    *   当用户点击支付时，服务端会以“订单号”为Key加**分布式锁**（Redisson）。
    *   这防止了用户疯狂点击按钮导致的并发请求，确保同一时刻只有一个请求能处理该订单的支付逻辑。
2.  **幂等性检查（防重复创建）：**
    *   在锁内部，代码会先查询数据库。
    *   如果发现该订单已经生成了有效的支付链接/二维码且未过期，直接返回已存在的二维码数据，而不会再次请求微信/支付宝接口。
    *   这样既解决了二维码不展示的问题（重试即可获取），又避免了重复调用第三方接口造成的资源浪费或数据混乱。

### 6. 支付渠道可能是多样的，你们的代码是怎么设计的？如果需要停用某个渠道，但是不重启服务，怎么能做到？

**代码设计：**
采用了**工厂模式 + 策略模式**（参考 `day03` 第4.4节 `HandlerFactory`）：
1.  **接口抽象：** 定义统一的 `NativePayHandler` 接口（包含下单、查单、退款等方法）。
2.  **具体实现：** 针对支付宝（`AliNativePayHandler`）和微信（`WechatNativePayHandler`）分别实现该接口。
3.  **注解标记：** 自定义 `@PayChannel(type = PayChannelEnum.ALI_PAY)` 注解标记在实现类上。
4.  **工厂获取：** `HandlerFactory` 在启动时扫描Bean，根据注解将其存入Map中。业务调用时，根据传入的渠道类型（ALIPAY/WECHAT）从工厂获取对应的Handler实现类。

**动态停用渠道（不重启）：**
1.  **数据库配置：** 我们有一张 `sl_pay_channel` 表，其中有一个字段 `enable_flag`（是否有效）。
2.  **读取逻辑：** 每次进行支付请求时，代码会先根据商户ID和渠道标识去数据库（或Redis缓存）查询该渠道的配置信息。
3.  **动态生效：** 如果管理员在后台将某个渠道的 `enable_flag` 改为禁用。下一次用户发起支付请求时，Service层查询该配置发现已被禁用，会直接抛出“渠道不可用”的异常，从而实现不重启服务即可停用渠道。

### 7. 支付成功的通知你们是怎处理的？有考虑幂等性吗？如何确保请求是真正来源支付平台的？如果通知没有接收到，你们怎么处理的？

**处理流程：**
我们采用**异步通知（回调） + 主动轮询**的双重保障机制。

1.  **验签（确保安全性）：**
    *   收到回调请求后，第一步是使用官方SDK进行**验签**（Alipay使用公钥验签，WeChat使用平台证书验签）。只有验签通过，才认为是合法的支付平台请求，防止伪造数据。
2.  **幂等性处理：**
    *   通过Redis分布式锁（Key为交易单号）防止回调并发。
    *   获取锁后，查询本地交易单状态。如果状态已经是“已支付/已结算”（`YJS`），则直接返回成功，**不进行后续业务逻辑**。
3.  **业务逻辑：**
    *   如果状态未支付，则更新交易单状态为“已支付”，清空二维码字段，并发送MQ消息通知订单服务、物流服务等下游系统。
4.  **通知没接收到（兜底方案）：**
    *   我们引入了 **XXL-JOB** 进行定时任务调度（参考 `day03` 第6章）。
    *   每隔一段时间（如每分钟）查询数据库中状态为“付款中”且创建时间超过一定阈值的订单。
    *   主动调用微信/支付宝的“查单接口”同步状态。如果查到已支付，则执行与回调相同的成功逻辑。

### 8. xxl-job的分片式广播用过吗？他是如何确保任务节点处理不重复数据的？

**用过。** 在支付状态同步和退款状态同步的定时任务中使用了**分片广播**路由策略（参考 `day03` 第6.3节）。

**确保不重复处理的原理：**
1.  **分片参数：** XXL-JOB调度中心在触发任务时，会给集群中的每个执行器节点下发两个参数：
    *   `shardIndex`：当前节点的索引（从0开始）。
    *   `shardTotal`：总的分片数量（即执行器节点的总数）。
2.  **取模逻辑：** 在业务代码中，我们利用这两个参数对数据进行过滤。
    *   SQL查询时或者在代码遍历集合时，使用公式：`ID % shardTotal == shardIndex`。
    *   例如：有3个节点（Total=3），节点0只处理 `ID % 3 == 0` 的数据，节点1处理 `ID % 3 == 1` 的数据，以此类推。
3.  **结果：** 每个节点只处理属于自己分片的那部分数据，互不干扰，从而实现了并行处理且不重复。

### 1. 细节: 如何保证订单幂等性?
**答：**
在支付系统中，幂等性是指同一个订单发起的多次支付请求，最终只会有一次有效的业务处理。我们在生成交易单（`idempotentCreateTrading`）时通过以下逻辑保证：
1.  **查询现有记录：** 根据业务订单号（`ProductOrderNo`）查询数据库。
2.  **状态判断：**
    *   若状态为**已结算（YJS）**：说明已支付成功，直接返回或抛出“已支付”异常，不再受理。
    *   若状态为**付款中（FKZ）**：
        *   **同渠道：** 如果支付渠道未变（如上次是支付宝，这次也是），直接返回原有的交易单信息（含二维码），不重新生成。
        *   **切渠道：** 如果用户切换了渠道（如支付宝换微信），则**保留原主键ID**，但**重新生成交易流水号（TradingOrderNo）**，并在数据库更新渠道信息，然后去第三方下单。
3.  **分布式锁：** 在整个处理过程外层加分布式锁（基于订单号），防止高并发下的重复创建。

### 2. 细节: 抽取支付通用服务的作用?
**答：**
1.  **解耦合：** 将支付逻辑与具体的业务逻辑（如运费支付、会员充值、违约金支付）剥离，业务方只需要调用统一接口，不需要关心对接支付宝还是微信的细节。
2.  **复用性：** 支付微服务作为一个独立的基础服务，可以被系统中的多个上游服务（如订单服务、营销服务）调用，避免重复造轮子。
3.  **统一管理：** 方便对支付渠道账号（AppID、密钥）进行统一配置和管理，也方便统一进行安全加固和日志监控。

### 3. 细节: 交易服务设计 -> 交易单表的作用?
**答：**
`sl_trading` 表是连接**内部业务订单**和**外部第三方支付**的桥梁。
*   **记录映射关系：** 它记录了系统的业务订单号（`ProductOrderNo`）和发往第三方的交易流水号（`TradingOrderNo`）的对应关系。
*   **状态追踪：** 独立记录支付状态（付款中、已结算、取消等），不干扰业务订单本身的状态流转。
*   **数据落地：** 存储支付金额、渠道、二维码信息、第三方返回的流水号等，用于后续的对账和退款依据。

### 4. 细节: 支付服务涉及哪几张表？
**答：**
主要涉及以下核心表（参考 `sl-express-ms-trade-service`）：
1.  **`sl_pay_channel`（支付渠道表）：** 存储支付宝、微信的商户号、AppID、公钥、私钥、回调地址等配置信息。
2.  **`sl_trading`（交易单表）：** 核心业务表，记录每一笔支付请求的详细信息（金额、状态、单号、二维码数据）。
3.  **`sl_refund_record`（退款记录表）：** 记录退款请求的流水号、金额、退款状态及第三方返回的退款单号。

### 6. 细节: 支付功能为什么要加分布式锁?
**答：**
*   **防止重复下单：** 用户点击支付按钮时可能会手抖或网络延迟导致重复提交，分布式锁能确保同一订单号在同一时间只有一个线程在处理创建交易单的逻辑。
*   **防止并发回调：** 支付成功后，支付宝/微信可能会多次发送异步通知（重试机制）。分布式锁能确保只有一个回调请求在更新数据库状态，避免数据错乱。
*   **解决幻读/覆盖：** 在切换支付渠道或退款时，防止并发操作导致查询到的状态不一致，从而覆盖了正确的数据。

### 7. 细节: 订单有哪些状态?
**答：**
根据 `TradingStateEnum` 枚举，主要状态包括：
*   **FKZ（付款中）：** 已生成二维码，等待用户扫码。
*   **YJS（已结算/支付成功）：** 用户已付款，第三方已确认。
*   **QXDD（取消订单/交易关闭）：** 超时未支付或全额退款。
*   **MD（免单）：** 金额为0的情况。
*   **GZ（挂账）：** 月结用户的特殊状态。

### 8. 细节: 如何得知支付的结果？项目中是如何处理的？ (异步通知，主动轮询 而且要详细说明)
**答：**
项目中采用了**“异步通知优先 + 主动轮询兜底”**的双重保障机制。
1.  **异步通知（回调）：**
    *   支付平台（支付宝/微信）在用户支付成功后，会POST请求我们预留的 `notify_url`。
    *   我们接收到请求后，**先验签**确保安全，然后获取分布式锁，检查本地订单状态。如果未支付，则更新为“已结算”并响应“Success/200”给支付平台。
2.  **主动轮询（XXL-JOB）：**
    *   因为网络原因回调可能会丢失。
    *   我们配置了 **XXL-JOB 分片广播任务**。
    *   定时扫描数据库中状态为“FKZ（付款中）”且创建时间超过一定阈值的订单。
    *   调用支付平台的**查单接口**（`alipay.trade.query` / `/v3/pay/transactions/out-trade-no/{}`），根据返回结果同步本地数据库状态。

### 9. 细节: 订单金额字段类型?
**答：**
*   **Java端：** 使用 `java.math.BigDecimal`。
    *   *原因：* `double` 和 `float` 是浮点数，在进行加减乘除运算时会丢失精度（如0.1+0.2!=0.3），涉及金钱必须使用 `BigDecimal`。
*   **数据库端：** 使用 `DECIMAL` 类型（如 `DECIMAL(10, 2)` 或更长精度）。

### 10. 细节: 如何保证支付安全? （重点 验签!! ）
**答：**
最核心的是**数字签名（Signature）与验签（Verify）**。
*   **请求时：** 我们使用自己的**私钥**对请求参数进行签名，支付平台用我们的公钥验签，确保请求是我们发出的且参数未被篡改。
*   **回调时（重点）：**
    *   支付平台发来通知时，会携带签名（sign）。
    *   **支付宝：** 我们使用支付宝的**公钥**，调用SDK的 `Factory.Payment.Common().verifyNotify(param)` 方法进行验签。
    *   **微信：** 我们使用微信的**平台公钥证书**，结合HTTP头中的 `Wechatpay-Signature` 等信息进行验签。
    *   只有验签通过，才信任该回调内容，否则直接丢弃，防止黑客伪造支付成功通知。

### 12. 细节: 支付接口需要传递哪些参数，调哪个接口，返回哪个值?
**答：**
以**扫码支付（Native Pay）**为例：
*   **传递参数：**
    *   `out_trade_no`：我们系统的唯一交易单号。
    *   `total_amount`：支付金额（支付宝单位元，微信单位分）。
    *   `subject`/`description`：商品标题/描述。
    *   `notify_url`：异步通知地址。
*   **调用接口：**
    *   支付宝：调用 `alipay.trade.precreate`。
    *   微信：调用 `/v3/pay/transactions/native`。
*   **返回值：**
    *   支付宝：返回 `qr_code` 字符串（二维码链接）。
    *   微信：返回 `code_url` 字符串。

### 13. 细节: 如何生成支付二维码?
**答：**
1.  服务端调用第三方支付接口，获取到支付链接字符串（如 `qr_code` 或 `code_url`）。
2.  服务端将该字符串返回给前端。
3.  **前端**（客户端/小程序）使用JS库（如 `qrcode.js`）将字符串转换为二维码图片展示。
    *   *好处：* 减轻服务端生成图片的CPU压力，节省网络带宽。

### 14. 细节: 如何动态的根据支付方式选择对应的支付?
**答：**
使用了**工厂模式 + 策略模式**。
1.  定义统一接口 `NativePayHandler`。
2.  创建两个实现类 `AliNativePayHandler` 和 `WechatNativePayHandler`。
3.  使用自定义注解 `@PayChannel(type=ALI_PAY)` 标记实现类。
4.  创建一个 `HandlerFactory`，在Spring启动时扫描Bean并将它们存入 `Map<ChannelEnum, Handler>`。
5.  运行时根据前端传来的渠道参数（ALI_PAY/WECHAT_PAY），从工厂Map中获取对应的Handler进行调用。

### 15. 细节: 对接支付宝和微信有哪些不同?
**答：**
1.  **SDK差异：** 支付宝提供了封装良好的 `EasySDK`；微信官方SDK较轻量，项目中是基于 `Apache HttpClient` 进行了二次封装。
2.  **加密方式：** 支付宝主要使用 RSA2；微信 V3 接口使用 AES-256-GCM 加密回调数据，且需要证书管理。
3.  **单位差异：** 支付宝金额单位是“元”（字符串或小数）；微信金额单位是“分”（整数）。
4.  **接口返回：** 支付宝返回 `qr_code`；微信返回 `code_url`。
5.  **回调验签：** 支付宝通过SDK方法直接验签；微信需要先解密请求体，再验证证书签名。

### 16. 细节: 项目中是否有对账功能?
**答：**
文档中提到了支付宝/微信的**下载账单API**（如 `alipay.data.dataservice.bill.downloadurl.query`），但在当前的业务代码展示中，主要聚焦于支付和退款流程。
*   *实际生产建议：* 会有一个独立的对账模块（T+1），每天下载第三方的账单文件，解析后与本地数据库的 `sl_trading` 表进行比对。如果发现本地有记录但第三方没记录（或反之、或金额不一致），则生成差异表进行人工或自动调账。

### 17. 细节: 用户支付的钱打到了哪里?
**答：**
钱是直接打到了**企业注册的支付宝/微信商户账户**中。
*   配置在 `sl_pay_channel` 表中的 AppID 和 商户号（MchID） 决定了资金的流向。
*   通常是 T+1 结算到企业的对公银行账户，或者停留在商户平台的余额中等待提现。

### 18. 细节: 如何实现一笔交易，分多批次退款?
**答：**
1.  **区分单号：** 必须区分“原交易单号（`out_trade_no`）”和“退款请求号（`out_request_no`）”。
2.  **多次请求：** 对同一笔 `out_trade_no` 可以发起多次退款请求。
3.  **唯一标识：** 每次退款时，生成一个新的 `refund_no`（退款请求号）传给支付平台，用于标识这笔具体的退款操作。
4.  **金额校验：** 支付平台会校验累计退款金额是否超过订单总金额。如果未超过，则退款成功；如果超过，则报错。
5.  **本地记录：** 每次退款都在 `sl_refund_record` 表中新增一条记录，关联同一个交易单。
