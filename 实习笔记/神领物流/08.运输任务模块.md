
# 面试连环问回答

## 1. 能说一下xxl-job的分片式调度是在什么场景下使用的吗？这样做的好处是什么？

**回答：**

在我们的智能调度系统中，**XXL-JOB 的分片广播调度主要用于“生成运输任务”的环节**。

- **业务场景**：
    
    调度是站在“车辆”的角度进行的。我们需要查询当前所有可用状态的车辆计划，根据车辆的起始和终点机构，去 Redis 中拉取对应的运单进行匹配。如果不使用分片，单机处理成千上万辆车的调度计算，效率会非常低且容易造成阻塞。
    
- **具体实现**：
    
    在代码中，我们利用 `XxlJobHelper.getShardIndex()` 获取当前分片序号，`XxlJobHelper.getShardTotal()` 获取分片总数。然后调用 `truckPlanFeign.pullUnassignedPlan(shardTotal, shardIndex)`，让每个执行器节点只拉取符合自己分片规则的那一部分车辆计划数据。
    
- **这样做的好处**：
    
    主要目的是为了**并行多处理车辆，提升调度处理效率**。通过集群部署，将庞大的车辆计算任务分散到不同的服务器上同时执行，极大地缩短了生成运输任务的时间。

## 2. 不同的运单流转节点是不一样的，你们如何将运单合并？如何确保Redis的高可用？

**回答：**

**关于运单合并：**

运单合并的核心逻辑是**基于“相同转运节点”进行分组**，并结合**车辆运力**进行计算。

1. **分组存储**：在前面的业务流程中，我们已经将具有相同起始机构（`startOrganId`）和目的机构（`endOrganId`）的运单，写入到了同一个 Redis 队列（List结构）中。
    
2. **调度计算**：
    
    - 调度器首先获取车辆计划，得到车辆的起始和结束机构 ID，从而锁定对应的 Redis Key。
        
    - 使用 `Redisson` 获取分布式锁，保证同一时间只有一个线程操作该车辆的调度。
        
    - 通过递归的方式（`executeTransportTask` 方法）从 Redis 队列中 `rightPop` 取出运单。
        
3. **运力校验**：
    
    - 每取出一个运单，系统会计算当前累计的重量和体积。
        
    - 将累计值与车辆的**最大允许载重**（`AllowableLoad`）和**最大允许体积**（`AllowableVolume`）进行比对（通常会乘以一个装载系数，如 0.95）。
        
    - 如果没有超出运力，则合并成功；如果超出运力，则将该运单通过 `rightPush` 放回队列最右侧，确保运单处理顺序，并结束当前车辆的装载。
        

**关于 Redis 高可用：**

（注：文档中主要提及了 Redis 的使用方式，未详细展开集群架构，以下结合文档提及的技术点回答）

在代码中我们使用了 `RedissonClient` 来实现分布式锁（`getFairLock`）。这暗示了底层 Redis 至少是支持高并发和分布式环境的。在实际生产中，为了确保高可用，通常会采用 **Redis Cluster（集群模式）** 或 **Sentinel（哨兵模式）**，结合 Redisson 提供的功能，既保证了数据分片存储（应对海量运单队列），也保证了在某个节点宕机时能够自动故障转移，确保调度任务不中断。

## 3. 你们系统中，车辆、车次和路线之间是什么关系？车辆有司机数量限制吗？

**回答：**

**三者关系：**

这三者是层级包含关系，用于确定物流的运输网络。

1. **线路（Line）**：代表物理上的运输轨迹（例如：北京转运中心 -> 上海转运中心）。
    
2. **车次（Transport Trips）**：是依附于线路的，代表该线路上具体的发车排班计划（例如：北京-上海线路上的 8:00 发车班次）。
    
3. **车辆（Truck）**：是配置在具体车次上的物理载体。我们在系统中需要先选择线路，再添加车次，最后为车次关联具体的车辆。
    

**司机数量限制：**

**有明确限制。**

系统要求**一个车辆至少配置 2 个司机**。

这是为了满足长途运输的轮班需求（主驾和副驾），保障运输安全。在生成运输任务时，如果检测到司机列表为空或者不满足条件，会影响后续司机作业单的生成。在测试数据配置环节，我们也必须为车辆配置至少两个司机才能正常跑通流程。
