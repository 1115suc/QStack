# LangChain4j

## 一、聊天与语言模型

### 1.`ChatRequest` 与 `ChatResponse`

- 如果你想自定义请求（例如指定模型名称、temperature、tools、JSON schema 等），可以使用 chat(ChatRequest) 方法：

```java
    ChatRequest chatRequest = ChatRequest.builder()
        .messages(...)
        .modelName(...)
        .temperature(...)
        .topP(...)
        .topK(...)
        .frequencyPenalty(...)
        .presencePenalty(...)
        .maxOutputTokens(...)
        .stopSequences(...)
        .toolSpecifications(...)
        .toolChoice(...)
        .responseFormat(...)
        .parameters(...) // 你也可以一次性设置通用或提供商特定的参数
        .build();
    
    ChatResponse chatResponse = chatModel.chat(chatRequest);
```

### 2.`ChatMessage` 的类型

- `UserMessage`: 这是来自用户的消息
- `SystemMessage`: 这是系统消息，通常用于提供上下文信息
- `AiMessage`: 这是 AI 生成的消息，用于响应输入的消息
  - text()：文本内容
  - thinking()：推理/思考内容
  - toolExecutionRequests()：执行工具的请求
  - attributes()：额外属性，通常是提供商特定的
- `CustomMessage`: 这是一个自定义消息，可以包含任意属性 (目前仅 Ollama 支持)

### 3.多个`ChatMessage` 

- LLM 本质上是无状态的，它们不会维护对话的上下文状态，所以需要自行维护多个 'ChatMessage' 对话上下文。

```java
    UserMessage firstUserMessage = UserMessage.from("Hello, my name is Klaus");
    AiMessage firstAiMessage = model.chat(firstUserMessage).aiMessage(); // Hi Klaus, how can I help you?
    UserMessage secondUserMessage = UserMessage.from("What is my name?");
    AiMessage secondAiMessage = model.chat(firstUserMessage, firstAiMessage, secondUserMessage).aiMessage(); // Klaus
```

### 4. 多模态

`UserMessage` 支持以下五种 `Content` 类型：`TextContent`、`ImageContent`、`AudioContent`、`VideoContent`、`PdfFileContent`。

> ⚠️ 请注意，并非所有模型都支持所有类型的媒体。有些模型仅支持图像，而另一些模型正在扩展到 PDF、视频或音频的支持。

#### 4.1 文本内容 (`TextContent`)

`TextContent` 是最简单的 `Content` 形式，代表纯文本，包装一个 `String`。`UserMessage.from(TextContent.from("Hello!"))` 等价于 `UserMessage.from("Hello!")`。

#### 4.2 图像内容 (`ImageContent`)

`ImageContent` 可以和 `TextContent` 一起组合发送到 LLM：

```java
UserMessage userMessage = UserMessage.from(
    TextContent.from("Describe the following image"),
    ImageContent.from("https://example.com/cat.jpg")
);
ChatResponse response = model.chat(userMessage);
```

`ImageContent` 支持通过 URL 或 Base64 编码的方式提供图像数据。

#### 4.3 音频内容 (`AudioContent`)

`AudioContent` 与 `ImageContent` 类似，但用于表示音频内容。例如使用 Gemini 对音频进行转录：

```java
var model = VertexAiGeminiChatModel.builder()
    .project(PROJECT_ID)
    .location(LOCATION)
    .modelName("gemini-1.5-pro")
    .build();

ChatResponse response = model.chat(
    UserMessage.from(
        AudioContent.from("https://example.com/audio/sample.mp3"),
        TextContent.from("Write a transcription of this audio file")
    )
);
```

#### 4.4 视频内容 (`VideoContent`)

`VideoContent` 与 `ImageContent` 类似，但用于表示视频内容。

```java
ChatResponse response = model.chat(
    UserMessage.from(
        VideoContent.from("https://example.com/video/sample.mp4"),
        TextContent.from("Describe what is happening in this video")
    )
);
```

#### 4.5 PDF文件内容 (`PdfFileContent`)

`PdfFileContent` 与 `ImageContent` 类似，但用于表示 PDF 二进制文件内容。Gemini 可以直接获取 PDF 文档，无需中间的文本转录。这使你可以针对 PDF 文档提出问题，而且由于 Gemini 拥有非常大的上下文窗口，它能够分析非常大的文档，或同时分析多个文档，而无需实现自己的 RAG 系统。

```java
ChatResponse response = model.chat(
    UserMessage.from(
        PdfFileContent.from("https://example.com/paper.pdf"),
        TextContent.from("Give a summary of this paper")
    )
);
```

#### 4.6 在 AI Service 中使用多模态

在 AI Service 的方法签名中，可以将 `Content` 作为参数传入，支持 `TextContent`、`ImageContent`、`AudioContent`、`VideoContent`、`PdfFileContent`：

```java
// 单个 Content
String chat(@UserMessage String userMessage, @UserMessage Content content);
String chat(@UserMessage String userMessage, @UserMessage ImageContent image);

// 多个 Content
String chat(@UserMessage String userMessage, @UserMessage List<Content> contents);

// 多种类型混合
String chat(@UserMessage AudioContent audio, @UserMessage ImageContent image);
```

AI Service 会按照参数声明的顺序将所有内容放入最终的 `UserMessage` 中。

### 5. 聊天记忆

#### 5.1 Memory 与 History 的区别

`History`（历史）保持用户和 AI 之间的所有消息完好无损，`History` 是用户在 UI 中看到的内容，它代表了实际说过的内容。`Memory`（记忆）保留一些信息，呈现给 LLM 使其表现得好像"记住"了对话。`Memory` 与 `History` 有很大不同。根据所使用的记忆算法，它可以以各种方式修改历史：淘汰某些消息、汇总多条消息、从消息中删除不重要的细节、向消息注入额外信息（例如用于 RAG）或指令（例如用于结构化输出）等。

LangChain4j 目前只提供 "memory"，不提供 "history"。如果需要保持完整的历史记录，请手动处理。

#### 5.2 ChatMemory 的两种实现

`ChatMemory` 抽象提供了多种开箱即用的实现。`ChatMemory` 可以作为独立的低级组件使用，也可以作为 AI Services 等高级组件的一部分使用。

**（1）MessageWindowChatMemory** — 基于消息数量的滑动窗口

`MessageWindowChatMemory` 作为滑动窗口运行，保留最近的 N 条消息并淘汰不再适合的旧消息。但由于每条消息可能包含不同数量的 token，`MessageWindowChatMemory` 主要适用于快速原型开发。

```java
ChatMemory chatMemory = MessageWindowChatMemory.withMaxMessages(10);
```

**（2）TokenWindowChatMemory** — 基于 Token 数量的滑动窗口

`TokenWindowChatMemory` 也作为滑动窗口运行，但关注于保持最近的 N 个 token，并根据需要淘汰旧消息。消息是不可分割的，如果一条消息不适合，它将被完全淘汰。`TokenWindowChatMemory` 需要一个 `TokenCountEstimator` 来计算每条 `ChatMessage` 中的 token 数。

```java
ChatMemory chatMemory = TokenWindowChatMemory.builder()
    .maxTokens(1000)
    .tokenCountEstimator(new OpenAiTokenCountEstimator("gpt-4o-mini"))
    .build();
```

#### 5.3 淘汰策略的特殊处理

如果一个包含 `ToolExecutionRequests` 的 `AiMessage` 被淘汰，后续的孤立 `ToolExecutionResultMessage` 也会被自动淘汰，以避免某些 LLM 提供商（如 OpenAI）禁止发送孤立 `ToolExecutionResultMessage` 的问题。

`MessageWindowChatMemory` 在淘汰最旧的消息时，会始终保留 `SystemMessage`（指令），以确保 bot 不会忘记它的角色设定。

#### 5.4 持久化

默认情况下，`ChatMemory` 实现将 `ChatMessage` 存储在内存中。如果需要持久化，可以实现自定义的 `ChatMemoryStore` 将 `ChatMessage` 存储在你选择的任何持久化存储中：

```java
class PersistentChatMemoryStore implements ChatMemoryStore {

    @Override
    public List<ChatMessage> getMessages(Object memoryId) {
        // TODO: 通过 memory ID 从持久化存储中获取所有消息
        // 可以使用 ChatMessageDeserializer.messagesFromJson(String) 辅助方法
    }

    @Override
    public void updateMessages(Object memoryId, List<ChatMessage> messages) {
        // TODO: 通过 memory ID 更新持久化存储中的所有消息
        // 可以使用 ChatMessageSerializer.messagesToJson(List) 辅助方法
    }

    @Override
    public void deleteMessages(Object memoryId) {
        // TODO: 通过 memory ID 删除持久化存储中的所有消息
    }
}
```

`updateMessages()` 方法在每次将新 `ChatMessage` 添加到 `ChatMemory` 时被调用。这通常在与 LLM 的每次交互中发生两次：一次是添加新的 `UserMessage`，一次是添加新的 `AiMessage`。`updateMessages()` 方法应更新与给定 memory ID 关联的所有消息。

请注意，从 `ChatMemory` 中淘汰的消息也会从 `ChatMemoryStore` 中淘汰。

使用方式：

```java
ChatMemory chatMemory = MessageWindowChatMemory.builder()
    .id("12345")
    .maxMessages(10)
    .chatMemoryStore(new PersistentChatMemoryStore())
    .build();
```

#### 5.5 为每个用户提供独立的聊天记忆

要为不同的用户或对话分配不同的记忆实例，可以使用 `@MemoryId` 注解方法参数。应用程序代码负责为每个用户或会话提供唯一的 memory ID，这使你能够控制记忆路由，并在单个服务实例内启用多会话支持。

```java
interface Assistant {
    String chat(@MemoryId int memoryId, @UserMessage String userMessage);
}

Assistant assistant = AiServices.builder(Assistant.class)
    .chatModel(model)
    .chatMemoryProvider(memoryId -> MessageWindowChatMemory.builder()
        .id(memoryId)
        .maxMessages(10)
        .build())
    .build();

// 用户1
String answerUser1 = assistant.chat(1, "Hello, my name is Klaus");
// 用户2
String answerUser2 = assistant.chat(2, "Hello, my name is Alice");
// 用户1 的后续对话（记忆独立）
String answer = assistant.chat(1, "What is my name?"); // Klaus
```

#### 5.6 为每个用户提供持久化聊天记忆

通过添加 `ChatMemoryProvider` 并使用 `@MemoryId`，可以将无状态的 LLM 转换为有状态的对话伙伴：

```java
@Bean
public ChatMemoryProvider chatMemoryProvider(PersistentChatMemoryStore store) {
    return memoryId -> MessageWindowChatMemory.builder()
        .id(memoryId)
        .maxMessages(20)
        .chatMemoryStore(store)    // <--- 连接持久化存储
        .build();
}
```

`ChatMemory` 充当短期记忆，`@MemoryId` 确保线程安全和用户隔离，淘汰策略防止超出上下文窗口。

### 6. 模型参数

根据你选择的模型和提供商，可以调整许多参数来定义：**模型的输出**——生成内容的创造性或确定性水平、生成内容的数量等；**连接性**——基本 URL、授权密钥、超时、重试、日志记录等。

通常，你可以在模型提供商的网站上找到所有参数及其含义。

#### 6.1 使用 Builder 模式设置参数（纯 Java）

```java
OpenAiChatModel model = OpenAiChatModel.builder()
    .apiKey(System.getenv("OPENAI_API_KEY"))
    .modelName("gpt-4o-mini")
    .temperature(0.3)
    .timeout(ofSeconds(60))
    .logRequests(true)
    .logResponses(true)
    .build();
```

#### 6.2 使用 `ChatRequestParameters` 设置默认参数

也可以使用 `ChatRequestParameters` 或提供商特定的 `OpenAiChatRequestParameters` 来指定默认聊天请求参数：

```java
ChatModel model = OpenAiChatModel.builder()
    .apiKey(System.getenv("OPENAI_API_KEY"))
    .defaultRequestParameters(OpenAiChatRequestParameters.builder()
        .modelName("gpt-4o-mini")
        .build())
    .build();
```

#### 6.3 Spring Boot 中配置

如果使用 Spring Boot starter，可以在 `application.properties` 文件中配置模型参数：

```properties
langchain4j.open-ai.chat-model.api-key=${OPENAI_API_KEY}
langchain4j.open-ai.chat-model.model-name=gpt-4o-mini
langchain4j.open-ai.chat-model.temperature=0.3
langchain4j.open-ai.chat-model.log-requests=true
langchain4j.open-ai.chat-model.log-responses=true
```

#### 6.4 常见参数说明

| 参数 | 说明 |
|---|---|
| `temperature` | 采样温度，介于 0 到 2 之间。较高的值（如 0.8）使输出更随机，较低的值（如 0.2）使输出更集中和确定性。 |
| `maxOutputTokens` | 聊天补全中可以生成的最大 token 数量。 |
| `frequencyPenalty` | 介于 -2.0 到 2.0 之间的数字。正值会根据新 token 在文本中的现有频率来惩罚它们，从而降低模型重复相同内容的可能性。 |
| `topP` | 核采样参数，模型考虑概率前 top_p 的 token 结果 |
| `modelName` | 要使用的模型名称 |
| `logRequests` / `logResponses` | 是否记录请求/响应日志（用于调试） |

### 7. 响应流式传输

LLM 每次生成一个 token 的文本，因此许多 LLM 提供商提供了一种逐 token 流式传输响应的方式，而不是等待整个文本生成完毕。这显著改善了用户体验，因为用户无需等待未知的时间，几乎可以立即开始阅读响应。

#### 7.1 `StreamingChatModel` 接口

对于 `ChatModel` 和 `LanguageModel` 接口，有对应的 `StreamingChatModel` 和 `StreamingLanguageModel` 接口。它们具有类似的 API，但可以流式传输响应。它们接受 `StreamingChatResponseHandler` 接口的实现作为参数。

#### 7.2 `StreamingChatResponseHandler` 的回调方法

当下一个部分文本响应生成时：调用 `onPartialResponse(String)` 或 `onPartialResponse(PartialResponse, PartialResponseContext)`。根据 LLM 提供商的不同，部分响应文本可以由单个或多个 token 组成。例如，你可以在 token 可用时立即将其发送到 UI。

当下一个部分思考/推理文本生成时：调用 `onPartialThinking(PartialThinking)` 或 `onPartialThinking(PartialThinking, PartialThinkingContext)`。

当下一个部分工具调用生成时：调用 `onPartialToolCall(PartialToolCall)` 或 `onPartialToolCall(PartialToolCall, PartialToolCallContext)`。

当流式传输完成时：`ChatResponse` 对象包含完整的响应（`AiMessage`）及 `ChatResponseMetadata`。

当发生错误时：调用 `onError(Throwable error)`。

#### 7.3 完整示例

```java
StreamingChatModel model = OpenAiStreamingChatModel.builder()
    .apiKey(System.getenv("OPENAI_API_KEY"))
    .modelName("gpt-4o-mini")
    .build();

    model.chat("Tell me a joke", new StreamingChatResponseHandler() {

    @Override
    public void onPartialResponse(String partialResponse) {
        System.out.print(partialResponse);  // 逐 token 打印
    }

    @Override
    public void onCompleteResponse(ChatResponse completeResponse) {
        System.out.println("\n\nDone! Full response: " 
            + completeResponse.aiMessage().text());
    }

    @Override
    public void onError(Throwable error) {
        error.printStackTrace();
    }
});
```

#### 7.4 使用 Lambda 简化

使用 Lambda 来流式传输响应非常简单。只需使用 `onPartialResponse()` 静态方法和一个定义处理部分响应逻辑的 Lambda 表达式：

```java
import static dev.langchain4j.model.LambdaStreamingResponseHandler.onPartialResponse;

model.chat("Tell me a joke", onPartialResponse(System.out::print));
```

同时处理部分响应和错误：

```java
import static dev.langchain4j.model.LambdaStreamingResponseHandler.onPartialResponseAndError;

model.chat("Tell me a joke",
    onPartialResponseAndError(System.out::print, Throwable::printStackTrace));
```

#### 7.5 在 AI Service 中使用流式传输

在 AI Service 中，使用 `TokenStream` 作为返回类型即可实现逐 token 流式传输：

```java
interface Assistant {
    TokenStream chat(String message);
}

StreamingChatModel model = OpenAiStreamingChatModel.builder()
    .apiKey(System.getenv("OPENAI_API_KEY"))
    .modelName("gpt-4o-mini")
    .build();

Assistant assistant = AiServices.create(Assistant.class, model);

TokenStream tokenStream = assistant.chat("Tell me a joke");

CompletableFuture<ChatResponse> futureResponse = new CompletableFuture<>();

tokenStream
    .onPartialResponse(partialResponse -> System.out.print(partialResponse))
    .onCompleteResponse(futureResponse::complete)
    .onError(futureResponse::completeExceptionally)
    .start();    // 别忘了调用 start()!

futureResponse.join();   // 阻塞等待完成
```

#### 7.6 取消流式传输

如果你希望取消流式传输，可以在 `onPartialResponse`、`onPartialThinking` 或 `onPartialToolCall` 方法的上下文对象中获取 `StreamingHandle`，调用其 `cancel()` 方法来停止响应。
