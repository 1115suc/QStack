# Semaphore 学习笔记

## 一、核心概念

**Semaphore（信号量）**：JUC 包中的同步工具类，用于控制同时访问特定资源的线程数量，实现限流和并发控制。

### 核心机制
- **许可（Permits）**：维护一组虚拟许可，线程需获取许可才能执行
- ** acquire()**：获取许可，无许可时阻塞等待
- **release()**：释放许可，唤醒等待线程
- **可重用**：许可总数固定，可循环获取和释放

---

## 二、核心 API

```java
// 1. 创建信号量（permits: 许可数, fair: 是否公平）
Semaphore semaphore = new Semaphore(int permits);
Semaphore semaphore = new Semaphore(int permits, boolean fair);

// 2. 获取许可（阻塞）
semaphore.acquire();
semaphore.acquire(int permits); // 获取多个许可

// 3. 尝试获取许可（非阻塞）
boolean success = semaphore.tryAcquire();
boolean success = semaphore.tryAcquire(long timeout, TimeUnit unit);

// 4. 释放许可
semaphore.release();
semaphore.release(int permits); // 释放多个许可

// 5. 查询状态
int available = semaphore.availablePermits(); // 可用许可数
int waiting = semaphore.getQueueLength();     // 等待线程数
```

## 三、典型应用场景

### 场景 1：接口限流

```java
public class ApiRateLimiter {
private final Semaphore semaphore = new Semaphore(10); // 最多10个并发

    public void handleRequest() {
        try {
            semaphore.acquire(); // 获取许可
            try {
                // 处理请求
                doBusiness();
            } finally {
                semaphore.release(); // 释放许可
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}
```

### 场景 2：数据库连接池

```java
public class ConnectionPool {
private final Semaphore semaphore;
private final Queue<Connection> pool;

    public ConnectionPool(int maxSize) {
        this.semaphore = new Semaphore(maxSize);
        this.pool = new LinkedList<>();
        // 初始化连接
        for (int i = 0; i < maxSize; i++) {
            pool.add(createConnection());
        }
    }

    public Connection getConnection() throws InterruptedException {
        semaphore.acquire(); // 获取许可
        synchronized (pool) {
            return pool.poll();
        }
    }

    public void releaseConnection(Connection conn) {
        synchronized (pool) {
            pool.offer(conn);
        }
        semaphore.release(); // 释放许可
    }
}
```

### 场景 3：控制并发下载任务

```java
public class DownloadManager {
private final Semaphore semaphore = new Semaphore(3); // 最多3个并发下载

    public void download(String url) {
        try {
            semaphore.acquire();
            try {
                System.out.println(Thread.currentThread().getName() + " 开始下载: " + url);
                Thread.sleep((long) (Math.random() * 5000));
                System.out.println(Thread.currentThread().getName() + " 下载完成: " + url);
            } finally {
                semaphore.release();
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }

    public static void main(String[] args) {
        DownloadManager manager = new DownloadManager();
        String[] urls = {"url1", "url2", "url3", "url4", "url5"};

        for (String url : urls) {
            new Thread(() -> manager.download(url), "Download-" + url).start();
        }
    }
}
```

### 场景 4：生产者-消费者模式

```java
public class ProducerConsumer {
private final Semaphore emptySlots;   // 空位信号量
private final Semaphore fullSlots;    // 满位信号量
private final Queue<Integer> buffer = new LinkedList<>();
private final int capacity = 5;

    public ProducerConsumer() {
        this.emptySlots = new Semaphore(capacity); // 初始有空位
        this.fullSlots = new Semaphore(0);         // 初始无数据
    }

    public void produce(int item) throws InterruptedException {
        emptySlots.acquire(); // 等待空位
        synchronized (buffer) {
            buffer.offer(item);
            System.out.println("生产: " + item);
        }
        fullSlots.release(); // 增加满位
    }

    public int consume() throws InterruptedException {
        fullSlots.acquire(); // 等待数据
        int item;
        synchronized (buffer) {
            item = buffer.poll();
            System.out.println("消费: " + item);
        }
        emptySlots.release(); // 增加空位
        return item;
    }
}
```

## 四、公平 vs 非公平

```java
// 公平信号量：FIFO 顺序获取许可
Semaphore fairSem = new Semaphore(5, true);

// 非公平信号量：允许插队（默认，性能更好）
Semaphore unfairSem = new Semaphore(5, false);
```

| 对比项 | 公平模式 | 非公平模式 |
|--------|---------|-----------|
| **获取顺序** | 严格按 FIFO | 允许插队 |
| **性能** | 较低（频繁上下文切换） | 较高 |
| **饥饿问题** | 无 | 可能存在 |
| **适用场景** | 需要严格顺序 | 大多数场景 |

## 五、注意事项

### ⚠️ 常见陷阱

1. **忘记释放许可**

```java
// ❌ 错误：异常时未释放许可
semaphore.acquire();
doSomething(); // 如果抛异常，permit 泄漏
semaphore.release();

// ✅ 正确：在 finally 中释放
try {
semaphore.acquire();
doSomething();
} finally {
semaphore.release();
}
```

2. **重复释放许可**

```java
// ❌ 危险：release 次数多于 acquire，许可数会超过初始值
semaphore.release();
semaphore.release(); // permit 数异常增加

// ✅ 原则：acquire 和 release 必须成对出现
```
3. **一次性获取多个许可**

```java
// 获取 3 个许可，如果不足则阻塞
semaphore.acquire(3);

// 注意：要么全部获取成功，要么全部等待
// 不会部分获取
```

## 六、底层原理

### 6.1 AQS 实现

```
Semaphore 内部结构：
┌──────────────────────────┐
│  Sync (extends AQS)      │
│  - state = permits       │  ← 许可数
│  - FIFO 等待队列          │  ← 存储等待线程
└──────────────────────────┘
```

### 6.2 工作流程

1. **acquire()**：
   - CAS 尝试将 state 减 1
   - 如果 state >= 0，获取成功
   - 如果 state < 0，进入等待队列阻塞

2. **release()**：
   - 将 state 加 1
   - 唤醒等待队列中的线程

3. **公平模式**：
   - 检查队列中是否有等待线程
   - 有则排队，无则直接获取

### 6.3 源码简化版

```java
public class Semaphore {
private final Sync sync;

    public Semaphore(int permits) {
        sync = new NonfairSync(permits);
    }

    public void acquire() throws InterruptedException {
        sync.acquireSharedInterruptibly(1);
    }

    public void release() {
        sync.releaseShared(1);
    }

    abstract static class Sync extends AbstractQueuedSynchronizer {
        Sync(int permits) {
            setState(permits);
        }

        protected final boolean tryReleaseShared(int releases) {
            for (;;) {
                int current = getState();
                int next = current + releases;
                if (compareAndSetState(current, next)) {
                    return true;
                }
            }
        }
    }

    static final class NonfairSync extends Sync {
        protected int tryAcquireShared(int acquires) {
            return nonfairTryAcquireShared(acquires);
        }
    }

    static final class FairSync extends Sync {
        protected int tryAcquireShared(int acquires) {
            for (;;) {
                if (hasQueuedPredecessors()) {
                    return -1; // 有等待线程，返回失败
                }
                int available = getState();
                int remaining = available - acquires;
                if (remaining < 0 || compareAndSetState(available, remaining)) {
                    return remaining;
                }
            }
        }
    }
}
```

## 七、最佳实践

### ✅ 推荐做法

1. **始终在 finally 中 release()**
2. **优先使用非公平模式（性能更好）**
3. **合理设置许可数（根据系统资源）**
4. **使用 tryAcquire(timeout) 避免无限等待**

### ❌ 避免做法

1. ~~忘记 release()~~
2. ~~release 次数多于 acquire~~
3. ~~许可数设置过大失去限流意义~~
4. ~~在高并发场景使用公平模式~~

## 九、面试高频问题

### Q1: Semaphore 的核心作用是什么？

**A:** 控制同时访问资源的线程数量，实现限流和并发控制。

### Q2: Semaphore 的公平和非公平模式有什么区别？

**A:** 
- 公平：严格按 FIFO 顺序获取许可
- 非公平：允许插队，性能更好（默认）

### Q3: 如果线程获取许可后未释放会怎样？

**A:** 可用许可数永久减少，可能导致其他线程永久阻塞（许可泄漏）。

### Q4: Semaphore 和 ReentrantLock 的区别？

**A:** 
- Semaphore：允许多个线程同时访问（基于许可数）
- ReentrantLock：同一时刻只允许一个线程访问（独占锁）

### Q5: Semaphore 可以实现互斥锁吗？

**A:** 可以，初始化许可数为 1 时，等价于互斥锁。
