## 一、JVM 底层架构

### 1. JVM 定义

- Java 虚拟机（Java Virtual Machine），负责执行 Java 字节码。

### 2. JVM 运行位置

- 运行在操作系统之上
- 依赖特定操作系统版本实现“一次编译，到处运行”

### 3. JVM 逻辑架构

**内存组成：运行时数据区**
- **线程共享**：堆、方法区
- **线程私有**：虚拟机栈、本地方法栈、程序计数器

**周边组件**
- **类加载子系统**：将类加载到内存
- **执行引擎**：包含解释器、即时编译器（JIT）、垃圾收集器
- **本地库接口**：调用本地方法库

### 4. 解释器 vs 即时编译器（JIT）

- **解释器**：逐行解释字节码为机器指令并执行，统计热点代码
- **JIT 编译器**：热点代码达到阈值后编译为机器指令，存入 JIT 代码缓存，提升执行效率
- **前端编译器**：Java → 字节码
- **后端编译器**：字节码 → 机器码（JIT）

### 5. 程序计数器（PC 寄存器）

- 指向下一条要执行的指令地址
- 本地方法：值为 undefined
- 非本地方法：指向当前正在执行的指令

## 二、类加载器

### 1. 类加载过程

加载 → 链接（验证、准备、解析） → 初始化

### 2. 类加载器分类

- **启动类加载器**（C++ 实现）
- **扩展/平台类加载器**
- **应用程序/系统类加载器**
- **用户自定义类加载器**（继承 ClassLoader）

> 以上类加载器都是上下级关系

### 3. 双亲委派模型

- 自底向上检查类是否已加载
- 自顶向下尝试加载
- 优点：效率、安全。（沙箱机制）
- 目的：
	- 性能：避免重复加载
	- 安全性：避免核心类被修改

### 4. 打破双亲委派模型

- **重写 loadClass()**：打破委派机制
- **重写 findClass()**：不打破委派机制
- 示例：Tomcat 自定义类加载器实现应用隔离

## 三、方法区

### 1. 存储内容

- **类型信息、方法信息、域信息、运行时常量池、JIT 代码缓存、静态变量(jkd6及之前)**

### 2. JDK 版本变化
- **JDK 7**：静态变量、字符串常量池移入堆
- **JDK 8+**：方法区从 JVM 内存移至本地内存（元空间）

> 方法区是一个概念，永久代=元空间 是其实现

## 四、方法调用栈

### 1. 栈结构

- 每个方法调用对应一个栈帧
- 栈帧包含：局部变量表、操作数栈、动态链接、方法返回地址

### 2. 异常

- `StackOverflowError`：栈内存溢出，常见于递归无法结束

## 五、堆

### 1. 分代结构（JDK 8+）

- **新生代**（Young Generation）
  - 伊甸园区（Eden）
  - 幸存者区（Survivor，From 和 To）
- **老年代**（Old Generation）
- **元空间**（Metaspace，替代永久代）

### 2. 内存比例（默认）

- 新生代 : 老年代 = 1 : 2
- 伊甸园 : 幸存者区 = 8 : 1 : 1

### 3. 对象分配与回收流程

1. 新对象创建在伊甸园区
2. Eden 满 → 触发 Minor GC（复制算法）
3. 存活对象移至 To 区，年龄 +1
4. From 和 To 交换
5. 年龄 > 15 → 移入老年代
6. 老年代满 → 触发 Major GC / Full GC
7. 若仍存不下 → OOM

### 4. JVM 参数设置
| 参数 | 说明 |
|------|------|
| `-Xms` | 堆初始大小 |
| `-Xmx` | 堆最大大小（建议 = Xms） |
| `-Xmn` | 新生代大小 |
| `-XX:SurvivorRatio` | 伊甸园与幸存者区比例 |

### 5. OOM 类型

- `Java heap space`：堆溢出
- `PermGen space`：永久代溢出（JDK 7 及以前）
- `Metaspace`：元空间溢出（JDK 8 及以后）

### 6. 内存分析工具

- **JProfiler**：可视化分析堆内存
- **Arthas**：在线诊断工具

## 六、垃圾回收（GC）

### 1. GC 涉及区域

- 堆
- 方法区（类卸载）

### 2. 方法区回收条件（类卸载）

- 该类的所有实例被回收
- 加载该类的 ClassLoader 被回收
- 该类的 Class 对象无引用

### 3. 垃圾判定算法
| 算法 | 说明 |
|------|------|
| 引用计数法 | 简单，但无法解决循环引用 |
| 可达性分析法 | 从 GC Root 出发，标记存活对象 |

### 4. 垃圾收集算法
| 算法 | 特点 |
|------|------|
| 标记-清除 | 两次遍历，产生碎片 |
| 复制算法 | 效率高，适合新生代 |
| 标记-整理 | 适合老年代，避免碎片 |
| 分代收集 | 综合使用以上算法 |

### 5. GC 性能指标

- **吞吐量**：用户代码运行时间 / 总时间
- **停顿时间**：GC 暂停应用的时间

### 6. 引用类型
| 类型  | 回收时机  | 说明             |
| --- | ----- | -------------- |
| 强引用 | 不回收   | 普通 new 对象      |
| 软引用 | 内存不足时 | 适合缓存           |
| 弱引用 | GC 时  | 适合 WeakHashMap |
| 虚引用 | 任何时候  | 无法获取对象，用于监控 GC |

### 7. 垃圾收集器对比

| 收集器                     | 类型  | 特点              | 关注点        |
| ----------------------- | --- | --------------- | ---------- |
| Serial / Serial Old     | 单线程 | 简单，适合单核         | 停顿时间       |
| ParNew                  | 多线程 | Serial 的多线程版    | 停顿时间       |
| Parallel / Parallel Old | 并行  | 自适应，可调参数        | 吞吐量        |
| CMS                     | 并发  | 老年代，低延迟         | 停顿时间       |
| G1                      | 整堆  | Region 管理，可预测停顿 | 吞吐量 + 停顿时间 |

#### CMS 阶段

1. 初始标记（STW）
2. 并发标记
3. 重新标记（STW）
4. 并发清除

#### G1 阶段

1. 初始标记（STW）
2. 并发标记
3. 最终标记（STW）
4. 筛选回收（STW）
