# 🚀 MySQL事务全面指南

## 📖 第一部分：事务基础概念

### 1.1 什么是事务？🤔

#### 学习目标
- 理解事务的基本概念和应用场景

#### 核心概念
**事务**指的是逻辑上的一组操作，组成这组操作的各个单元要么全部成功，要么全部失败。

**事务作用**：保证在一个事务中多次操作数据库表中数据时，要么全都成功，要么全都失败。

#### 实际应用场景
💡 **电商转账案例**：
- 买家付款后，如果转账系统宕机，卖家未收到钱但买家已扣款 → 不合理
- 买家付款后撤销交易，但卖家已发货 → 不合理

![](img\mysql事务1.bmp)

> **说明**：在数据库中查询不会涉及到使用事务，都是增删改操作。

#### 💡 小结
事务是**多条SQL组合在一起完成某个功能**的机制。

## ⚙️ 第二部分：事务操作实践

### 2.1 手动提交事务 🛠️

#### 学习目标
- 掌握手动提交事务的使用方法

#### 事务操作方式
MySQL支持两种事务操作方式：
- ✅ **手动提交事务**：先开启，再提交
- 🔄 **自动提交事务**（默认）：执行一条SQL语句提交一次事务

#### 核心SQL语句
| SQL语句 | 描述 | 表情 |
|---------|------|------|
| `start transaction;` | 开启手动控制事务 | 🚪 |
| `commit;` | 提交事务 | ✅ |
| `rollback;` | 回滚事务 | ↩️ |

#### 操作流程图
![](img\事务01.png)

#### 准备测试数据
```sql
# 创建测试数据库和表
create database day04_db; 
use day04_db;

create table account(
	id int primary key auto_increment,
	name varchar(20),
	money double
);

# 初始化测试数据
insert into account values (null,'a',1000);
insert into account values (null,'b',1000);
```

#### 案例演示
**✅ 成功提交事务**：a给b转账100元
![](img\提交事务.bmp)

**↩️ 回滚事务**：a给b转账100元（失败）
![](img\回滚事务.bmp)

#### 关键要点
- 提交事务后，数据永久保存，无法回滚
- 回滚事务会撤销所有已执行的SQL语句
- 提交或回滚都代表结束当前事务

### 2.2 自动提交事务 ⚡

#### 学习目标
- 了解自动提交事务机制
- 掌握关闭自动提交的方法

#### 自动提交机制
MySQL默认每条DML语句都是一个单独的事务，执行完毕自动提交。

#### 查看和设置自动提交
```sql
# 查看当前autocommit模式
show variables like '%commit%';
```
![](img\自动提交事务.bmp)

```sql
# 关闭自动提交（临时生效）
set autocommit = 0;  -- 0:关闭  1:开启
```

#### 实践代码示例
```sql
-- 自动提交事务演示
update account set money=money-100 where name='a';
update account set money=money+100 where name='b';

-- 查看和设置自动提交
show variables like '%commit%';
set autocommit = 0;
```

#### 💡 重要说明
1. MySQL**默认自动提交**，执行一条SQL提交一次事务
2. `set autocommit = 0`只是**临时设置**，重新连接后恢复默认
3. 手动开启事务(`start transaction;`)会**覆盖自动提交设置**
4. 不手动开启事务时，每条SQL都是独立事务

## 🔍 第三部分：事务原理与特性

### 3.1 事务工作原理 🔧

![](img\事务19.png)

#### 原理说明
1. **临时日志文件**：用户登录后创建，保存事务状态
2. **无事务操作**：直接写入数据库，不使用日志文件
3. **开启事务**：所有写操作写入日志文件
4. **提交事务**：日志文件内容写入数据库
5. **回滚事务**：清空日志文件，不影响数据库

#### 事务操作总结表
| 操作 | MySQL语句 | 说明 |
|------|-----------|------|
| 手动开启事务 | `start transaction` | 🚪 开启事务 |
| 手动提交事务 | `commit` | ✅ 永久保存 |
| 手动回滚事务 | `rollback` | ↩️ 撤销操作 |
| 查询自动提交 | `show variables like '%commit%'` | 🔍 查看状态 |
| 设置手动提交 | `set autocommit = 0` | ⚙️ 关闭自动提交 |

### 3.2 事务的四大特性(ACID) 🎯

#### 学习目标
- 掌握事务的ACID特性

#### 四大特性详解

**🔒 隔离性 (Isolation)**
- 多个并发事务之间相互独立，互不干扰
- 例：a转账给b 和 c转账给d 两个事务互不影响

**💾 持久性 (Durability)**
- 事务提交后，对数据库的改变是永久性的
- 即使数据库异常重启，数据依然保持不变

**⚛️ 原子性 (Atomicity)**
- 事务中的操作要么全部成功，要么全部失败
- 不可分割的工作单位

**🔄 一致性 (Consistency)**
- 事务执行前后，数据库必须处于一致性状态
- 成功：所有变化生效；失败：回滚到原始状态

#### ACID特性总结表
| 特性 | 含义 | 重要性 |
|------|------|--------|
| 一致性 | 事务前后数据完整性保持一致 | 🌟🌟🌟🌟🌟 |
| 原子性 | 事务操作要么全成功，要么全失败 | 🌟🌟🌟🌟🌟 |
| 隔离性 | 并发事务之间相互隔离 | 🌟🌟🌟🌟 |
| 持久性 | 事务提交后改变永久保存 | 🌟🌟🌟🌟 |

## ⚠️ 第四部分：事务并发问题

### 4.1 并发访问的三大问题 🚨

#### 学习目标
- 理解事务并发访问可能引发的问题

#### 问题分类与说明

**💀 脏读 (Dirty Read)**
- **定义**：一个事务读取了另一个事务**未提交的数据**
- **严重程度**：⭐️⭐️⭐️⭐️⭐️（必须避免）
- **示意图**：![](img\脏读.bmp)

**🔄 不可重复读 (Non-repeatable Read)**
- **定义**：同一事务内多次读取的**数据内容不一致**
- **原因**：另一个事务执行了**UPDATE**操作
- **示意图**：![](img\不可重复读.bmp)

**👻 幻读 (Phantom Read)**
- **定义**：同一事务内多次读取的**数据数量不一致**
- **原因**：另一个事务执行了**INSERT或DELETE**操作
- **示意图**：![](img\幻读.bmp)

#### 问题对比表
| 问题类型 | 读取内容 | 其他事务操作 | 严重程度 |
|----------|----------|--------------|----------|
| 脏读 | 未提交数据 | 任何修改 | ⭐️⭐️⭐️⭐️⭐️ |
| 不可重复读 | 已提交数据 | UPDATE | ⭐️⭐️⭐️ |
| 幻读 | 已提交数据 | INSERT/DELETE | ⭐️⭐️⭐️ |

## 🛡️ 第五部分：事务隔离级别

### 5.1 MySQL隔离级别详解 🎚️

#### 学习目标
- 掌握MySQL的四种隔离级别及其应用

#### 隔离级别对比表
| 级别 | 名称 | 隔离级别 | 脏读 | 不可重复读 | 幻读 | 默认使用 |
|------|------|----------|------|------------|------|----------|
| 1 | 读未提交 | read uncommitted | ✅ | ✅ | ✅ | - |
| 2 | 读已提交 | read committed | ❌ | ✅ | ✅ | Oracle/SQL Server |
| 3 | 可重复读 | repeatable read | ❌ | ❌ | ✅ | **MySQL** |
| 4 | 串行化 | serializable | ❌ | ❌ | ❌ | - |

#### 安全性与性能权衡
- **安全性**：serializable > repeatable read > read committed > read uncommitted
- **性能**：serializable < repeatable read < read committed < read uncommitted

#### 💡 实际应用建议
- **脏读**：必须避免，逻辑错误
- **不可重复读/幻读**：时效性问题，可接受（根据业务需求）

### 5.2 隔离级别设置与实践 🔧

#### 查询当前隔离级别
```sql
-- 方法一：查看全局事务隔离级别
show variables like '%isolation%';

-- 方法二：查询当前隔离级别
select @@tx_isolation;
```

#### 设置隔离级别
```sql
-- 设置全局隔离级别（需要重新连接生效）
set global transaction isolation level 隔离级别;

-- 示例：
set global transaction isolation level read uncommitted;
set global transaction isolation level repeatable read;
```

#### 🚨 注意事项
MySQL 5.6+版本设置`read uncommitted`可能报错，需要额外设置：
```sql
# 设置session级别的BINLOG格式
SET SESSION binlog_format = 'MIXED';

# 查看当前BINLOG格式
select @@binlog_format;
```
