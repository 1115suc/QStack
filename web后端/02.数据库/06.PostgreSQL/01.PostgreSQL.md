# PostgreSQL 

---

## 一、为什么选择 PostgreSQL？

作为后端开发，选型时常见的对比是 MySQL vs PostgreSQL。简单总结：

| 维度 | MySQL | PostgreSQL |
|------|-------|------------|
| 定位 | 偏 OLTP，简单易上手 | 功能更完整，更接近 SQL 标准 |
| JSON 支持 | JSON 类型（功能有限） | JSONB（二进制存储，可索引，功能强大）|
| 数组类型 | ❌ 不支持 | ✅ 原生支持 |
| 窗口函数 | 支持（8.0+） | 支持且更完善 |
| 事务完整性 | 依赖存储引擎（InnoDB） | 全面支持 ACID |
| 并发控制 | MVCC（有部分锁竞争）| MVCC（无读锁，读写不阻塞）|
| 扩展性 | 一般 | 极强（PostGIS、TimescaleDB 等）|
| 大小写敏感 | 默认不敏感 | **标识符默认小写，字符串区分大小写** |

> 📌 **结论**：项目对数据类型复杂度要求高、需要强一致性、或有地理/时序需求时，优先考虑 PostgreSQL。

## 二、安装 & 快速启动

### 2.1 Docker 启动（推荐开发环境）

```bash
docker run -d \
  --name postgres \
  -e POSTGRES_USER=admin \
  -e POSTGRES_PASSWORD=123456 \
  -e POSTGRES_DB=mydb \
  -p 5432:5432 \
  postgres:16
```

### 2.2 连接数据库

```bash
# 进入容器内 psql
docker exec -it postgres psql -U admin -d mydb

# 或本机安装后直接使用
psql -h localhost -U admin -d mydb -p 5432
```

### 2.3 常用 psql 命令

```sql
\l          -- 列出所有数据库
\c mydb     -- 切换数据库
\dt         -- 列出当前 schema 的所有表
\d users    -- 查看 users 表结构
\du         -- 列出所有用户/角色
\q          -- 退出
```

## 三、核心概念

### 3.1 Schema（模式）

PostgreSQL 引入了 **Schema** 层级，结构为：

```
Database → Schema → Table
```

默认 Schema 是 `public`。可以通过 Schema 隔离不同业务模块的表：

```sql
-- 创建 Schema
CREATE SCHEMA order_service;

-- 在指定 Schema 下建表
CREATE TABLE order_service.orders (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL
);

-- 查询时需带 Schema 前缀（或设置 search_path）
SELECT * FROM order_service.orders;

-- 设置默认搜索路径
SET search_path TO order_service, public;
```

> 📌 **后端开发注意**：在 JDBC 连接串中可以指定 `currentSchema` 参数，避免每次写全路径。

```
jdbc:postgresql://localhost:5432/mydb?currentSchema=order_service
```

### 3.2 数据类型对照表（MySQL → PostgreSQL）

| MySQL 类型 | PostgreSQL 类型 | 说明 |
|-----------|----------------|------|
| `INT` | `INTEGER` / `INT4` | 4字节整型 |
| `BIGINT` | `BIGINT` / `INT8` | 8字节整型 |
| `TINYINT(1)` | `BOOLEAN` | 布尔值 |
| `AUTO_INCREMENT` | `SERIAL` / `BIGSERIAL` | 自增主键 |
| `VARCHAR(n)` | `VARCHAR(n)` / `TEXT` | 字符串（TEXT 无长度限制）|
| `DATETIME` | `TIMESTAMP` | 时间戳 |
| `JSON` | `JSON` / `JSONB` | JSONB 支持索引 |
| `DOUBLE` | `FLOAT8` / `NUMERIC` | 浮点/精确小数 |
| `BLOB` | `BYTEA` | 二进制数据 |

### 3.3 自增主键的写法

MySQL 习惯写 `AUTO_INCREMENT`，在 PostgreSQL 中有两种方式：

```sql
-- 方式1：SERIAL（传统方式）
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,  -- 等价于 BIGINT + 序列
    username VARCHAR(50) NOT NULL
);

-- 方式2：IDENTITY（PostgreSQL 10+，更符合 SQL 标准，推荐）
CREATE TABLE users (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    username VARCHAR(50) NOT NULL
);
```

> 📌 与 MyBatis-Plus 整合时，需要将主键策略设为 `IdType.AUTO`，并确保表使用了自增列。

## 四、DDL 语法差异与注意点

### 4.1 建表示例

```sql
CREATE TABLE users (
    id          BIGSERIAL PRIMARY KEY,
    username    VARCHAR(50)  NOT NULL UNIQUE,
    email       VARCHAR(100) NOT NULL,
    age         SMALLINT     CHECK (age >= 0 AND age <= 150),
    is_active   BOOLEAN      NOT NULL DEFAULT TRUE,
    created_at  TIMESTAMP    NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP    NOT NULL DEFAULT NOW(),
    deleted_at  TIMESTAMP    -- 软删除字段，NULL 表示未删除
);

COMMENT ON TABLE users IS '用户表';
COMMENT ON COLUMN users.username IS '用户名，唯一';
```

### 4.2 修改表结构

```sql
-- 新增列
ALTER TABLE users ADD COLUMN phone VARCHAR(20);

-- 修改列类型（PostgreSQL 比 MySQL 严格，需显式转换）
ALTER TABLE users ALTER COLUMN age TYPE INTEGER;

-- 设置列默认值
ALTER TABLE users ALTER COLUMN is_active SET DEFAULT TRUE;

-- 删除列
ALTER TABLE users DROP COLUMN phone;

-- 重命名列
ALTER TABLE users RENAME COLUMN email TO email_address;
```

### 4.3 字符串大小写敏感

```sql
-- PostgreSQL 字符串比较区分大小写！
SELECT * FROM users WHERE username = 'Admin';   -- 找不到 'admin'
SELECT * FROM users WHERE username = 'admin';   -- 正确

-- 如需不区分大小写，使用 ILIKE 或 LOWER()
SELECT * FROM users WHERE username ILIKE 'admin';
SELECT * FROM users WHERE LOWER(username) = LOWER('Admin');
```

## 五、DML 差异与常用写法

### 5.1 UPSERT（插入或更新）

MySQL 用 `ON DUPLICATE KEY UPDATE`，PostgreSQL 用 `ON CONFLICT`：

```sql
-- 存在则更新，不存在则插入
INSERT INTO users (username, email)
VALUES ('alice', 'alice@example.com')
ON CONFLICT (username)
DO UPDATE SET
    email = EXCLUDED.email,
    updated_at = NOW();

-- 存在则忽略
INSERT INTO users (username, email)
VALUES ('alice', 'alice@example.com')
ON CONFLICT (username) DO NOTHING;
```

### 5.2 RETURNING（插入后返回数据）

非常实用，省去一次额外查询：

```sql
-- 插入后直接返回生成的 id
INSERT INTO users (username, email)
VALUES ('bob', 'bob@example.com')
RETURNING id, created_at;

-- 更新后返回最新值
UPDATE users SET email = 'new@example.com'
WHERE id = 1
RETURNING id, email, updated_at;
```

> 📌 **后端开发重点**：MyBatis 中配合 `useGeneratedKeys` 或 `<selectKey>` 使用，Part 3 会详细讲解。

### 5.3 分页查询

```sql
-- PostgreSQL 分页写法（与 MySQL 一致）
SELECT * FROM users
WHERE is_active = TRUE
ORDER BY created_at DESC
LIMIT 10 OFFSET 0;   -- 第1页
```

### 5.4 字符串拼接

```sql
-- MySQL 用 CONCAT()，PostgreSQL 支持 || 运算符
SELECT 'Hello' || ' ' || 'World';   -- Hello World

-- 也可用 CONCAT()
SELECT CONCAT(first_name, ' ', last_name) AS full_name FROM users;
```

## 六、索引

### 6.1 常用索引类型

| 索引类型 | 适用场景 |
|---------|---------|
| B-Tree（默认）| 等值、范围查询，排序 |
| Hash | 等值查询（=），比 B-Tree 稍快，但不支持排序 |
| GIN | 全文搜索、JSONB、数组 |
| GiST | 地理数据（PostGIS）|
| BRIN | 超大表的范围查询（如时序数据）|

### 6.2 创建索引

```sql
-- 普通索引
CREATE INDEX idx_users_email ON users(email);

-- 唯一索引
CREATE UNIQUE INDEX idx_users_username ON users(username);

-- 复合索引
CREATE INDEX idx_users_active_created ON users(is_active, created_at DESC);

-- 部分索引（只对满足条件的行建索引，节省空间）
CREATE INDEX idx_users_active ON users(id)
WHERE is_active = TRUE;

-- JSONB 列上建 GIN 索引
CREATE INDEX idx_users_meta ON users USING GIN(meta);
```

### 6.3 查看执行计划

```sql
EXPLAIN SELECT * FROM users WHERE email = 'alice@example.com';

-- 实际执行并输出详细信息
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'alice@example.com';
```

## 七、事务

### 7.1 基本事务

```sql
BEGIN;

UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

-- 如果没有错误
COMMIT;

-- 如果有错误则回滚
ROLLBACK;
```

### 7.2 保存点（Savepoint）

```sql
BEGIN;

INSERT INTO orders (...) VALUES (...);

SAVEPOINT sp1;

INSERT INTO order_items (...) VALUES (...);

-- 如果 order_items 插入失败，只回滚到保存点
ROLLBACK TO SAVEPOINT sp1;

COMMIT;
```

### 7.3 事务隔离级别

```sql
-- 设置当前事务的隔离级别
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 可选级别（从低到高）：
-- READ COMMITTED（默认）
-- REPEATABLE READ
-- SERIALIZABLE
```

> 📌 PostgreSQL 默认隔离级别是 `READ COMMITTED`，与 MySQL 不同（MySQL 默认 `REPEATABLE READ`）。Spring Boot 整合时注意 `@Transactional` 的 `isolation` 参数。

## 八、权限管理

```sql
-- 创建用户
CREATE USER app_user WITH PASSWORD 'secure_password';

-- 授予数据库连接权限
GRANT CONNECT ON DATABASE mydb TO app_user;

-- 授予 Schema 使用权限
GRANT USAGE ON SCHEMA public TO app_user;

-- 授予表的读写权限
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;

-- 授予序列使用权限（自增主键依赖序列）
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO app_user;
```

## 小结

| 知识点 | 要点 |
|--------|------|
| Schema | 多了一层隔离，JDBC 连接串可配置 `currentSchema` |
| 自增主键 | 用 `BIGSERIAL` 或 `GENERATED ALWAYS AS IDENTITY` |
| UPSERT | 使用 `ON CONFLICT` 语法 |
| RETURNING | 插入/更新后直接返回数据，减少查询次数 |
| 字符串 | 大小写敏感，模糊匹配用 `ILIKE` |
| 事务隔离 | 默认 `READ COMMITTED`，与 MySQL 不同 |
| 索引 | JSONB/数组用 GIN 索引，部分索引节省空间 |
