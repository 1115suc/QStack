# MCP进阶篇

## 1. 进阶开发的核心变化

基础阶段关注 MCP 是什么；进阶阶段关注如何设计和实现 MCP Server。

进阶阶段要解决的问题：

- 工具参数如何设计
- 返回结果如何结构化
- 本地文件工具如何限制访问范围
- MCP Server 如何接入现有业务 API
- 如何让 Agent 使用 MCP 工具
- 如何处理错误、权限和日志

进阶 MCP 的核心是工具服务工程化。

## 2. 推荐项目结构

```text
mcp_server/
  main.py
  server.py
  config.py
  tools/
    notes_tool.py
    database_tool.py
    order_tool.py
  resources/
    notes_resource.py
    schema_resource.py
  prompts/
    rag_prompt.py
  services/
    note_service.py
    order_service.py
  security/
    permission.py
```

分层职责：

| 层 | 职责 |
| --- | --- |
| server | 注册 MCP tools、resources、prompts |
| tools | 定义可调用工具 |
| resources | 定义可读取资源 |
| prompts | 定义提示词模板 |
| services | 封装业务逻辑 |
| security | 权限、路径限制、参数校验 |

不要把文件读取、数据库访问、HTTP 请求和工具注册全部写在一个函数里。

## 3. Tool 设计进阶

MCP Tool 是给模型调用的接口，因此描述要比普通 API 更清楚。

坏设计：

```text
tool: query
description: 查询信息
input: text
```

好设计：

```text
tool: search_notes
description: 当用户需要在 Obsidian 技术笔记中查找相关内容时使用。
input:
  keyword: 搜索关键词，例如 FastAPI 依赖注入
  directory: 限制搜索目录，例如 python/Agent
  limit: 最多返回结果数量
```

Tool 设计 Checklist：

- 名称是否表达动作
- 描述是否说明使用场景
- 参数是否有类型和示例
- 是否限制返回数量
- 是否处理空结果
- 是否处理权限不足
- 是否有超时

## 4. 本地文件工具

本地文件工具最常见，也最容易出安全问题。

风险：

- 读取任意文件
- 泄露密钥
- 路径穿越
- 返回过长内容
- 读取二进制文件

安全策略：

- 配置 allowed roots
- 只允许读取指定后缀
- 规范化路径后检查是否在允许目录内
- 限制最大读取长度
- 对 `.env`、密钥文件、证书文件默认拒绝

伪代码：

```python
from pathlib import Path

ALLOWED_ROOT = Path("D:/Users/32147/Documents/Obsidian Vault/QStack").resolve()

def resolve_safe_path(relative_path: str) -> Path:
    target = (ALLOWED_ROOT / relative_path).resolve()
    if not str(target).startswith(str(ALLOWED_ROOT)):
        raise PermissionError("路径超出允许范围")
    if target.suffix.lower() not in {".md", ".txt"}:
        raise PermissionError("只允许读取 Markdown 或文本文件")
    return target
```

## 5. 数据库工具

数据库工具建议先做只读查询。

不推荐直接暴露：

```text
execute_sql(sql)
```

风险：

- SQL 注入
- 删除或修改数据
- 查询敏感字段
- 返回过多数据

推荐封装成业务语义工具：

```text
get_order_status(order_id)
list_user_orders(user_id, limit)
get_product_inventory(product_id)
```

如果必须做 SQL 分析工具：

- 只允许 SELECT
- 限制表白名单
- 限制返回行数
- 禁止访问敏感列
- 查询前做 explain 或人工确认

## 6. HTTP API 工具

业务 API 可以通过 MCP 标准化。

工具示例：

```text
get_order_status(order_id)
create_support_ticket(title, description, priority)
```

HTTP 工具要点：

- 设置 timeout
- 处理 4xx / 5xx
- 不把内部 token 返回给模型
- 对返回 JSON 做裁剪
- 记录 request_id
- 高风险接口默认 dry-run

返回结果示例：

```json
{
  "order_id": "A1001",
  "status": "已支付，等待发货",
  "updated_at": "2026-06-02"
}
```

## 7. Resource 设计

Resource 更适合静态或半静态上下文。

适合做 resource：

- 项目 README
- API schema
- 数据库表结构
- 知识库目录
- 配置说明

不适合做 resource：

- 会产生副作用的操作
- 需要复杂参数的查询
- 高风险业务动作

示例：

```text
resource: obsidian://python/Agent/03.LangChain
description: LangChain 学习笔记目录
mime_type: text/markdown
```

Resource 要点：

- URI 要稳定。
- 内容不要过长。
- 可以按目录、文件、业务对象拆分。
- 敏感资源要做权限控制。

## 8. Prompt 设计

MCP Prompt 可以沉淀团队通用任务模板。

示例：

```text
prompt: rag_answer_prompt
description: 基于检索上下文生成严谨答案
arguments:
  question: 用户问题
  context: 检索到的上下文
```

模板内容：

```text
你是一个严谨的知识库问答助手。
只能根据上下文回答问题。
如果上下文没有答案，回答“资料中没有提到”。

上下文：
{context}

问题：
{question}
```

Prompt 适合统一团队风格，避免每个项目重复写相似提示词。

## 9. MCP 接入 Agent

Agent 接入 MCP 的思路：

```text
连接 MCP Server
  -> 获取 tools/list
  -> 转换为 Agent 可用工具
  -> Agent 执行时调用 tools/call
```

接入后，Agent 可以使用 MCP Server 暴露的工具。

```text
用户：根据我的 LangChain 笔记总结 Agent 工程化要点
Agent -> search_notes
Agent -> read_note
Agent -> 总结回答
```

实践中要注意：

- 不要把所有 MCP 工具一次性暴露给 Agent。
- 根据用户权限过滤工具。
- 根据任务场景动态选择工具。
- 记录 MCP 工具调用日志。

## 10. 进阶阶段实践任务

实现一个 Obsidian MCP Server：

- 只允许访问当前笔记仓库
- `list_notes(directory)`
- `read_note(path)`
- `search_notes(keyword, directory, limit)`
- 暴露 `obsidian://python/Agent` resource
- 暴露 `summarize_note_prompt`
- 接入一个 Agent，让 Agent 可以查询你的学习笔记

进阶阶段目标是掌握 MCP Server 的工具设计、权限边界和 Agent 接入。
