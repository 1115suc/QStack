# MCP基础篇

## 1. MCP 解决什么问题

MCP 是 Model Context Protocol，目标是为 AI 应用提供统一的上下文和工具协议。

没有 MCP 时，不同 Agent 或客户端通常要重复封装工具：

```text
ChatGPT 工具封装一遍
Claude 工具封装一遍
Cursor 工具封装一遍
自研 Agent 工具再封装一遍
```

MCP 的思路是：把工具、资源和提示词封装成标准 MCP Server，让不同 MCP Client 都可以接入。

```text
AI 应用 / Agent
  -> MCP Client
  -> MCP Server
  -> 文件 / 数据库 / API / 业务系统
```

它解决的问题：

- 工具接口不统一
- 不同客户端重复开发工具
- 本地文件、数据库、API 难以标准接入
- Agent 与业务系统耦合严重
- 工具能力难以复用和治理

## 2. MCP 的核心角色

| 角色 | 说明 |
| --- | --- |
| MCP Host | 使用 MCP 的宿主应用，例如 AI IDE、聊天客户端、自研 Agent |
| MCP Client | Host 内部负责连接 MCP Server 的客户端 |
| MCP Server | 对外暴露 tools、resources、prompts |
| Tool | 模型可以调用的动作 |
| Resource | 模型可以读取的上下文资源 |
| Prompt | 可复用的提示词模板或工作流模板 |

可以这样理解：

```text
Host 是使用者
Client 是连接器
Server 是能力提供者
Tool 是动作
Resource 是资料
Prompt 是模板
```

## 3. MCP Server 提供什么

MCP Server 主要提供三类能力。

### 3.1 Tools

Tools 是模型可以调用的函数或动作。

示例：

```text
search_notes(keyword)
query_database(sql)
get_order_status(order_id)
create_ticket(title, description)
```

适合：

- 查询数据
- 调用 API
- 执行业务动作
- 检索知识库
- 操作文件

### 3.2 Resources

Resources 是可读取的上下文资料。

示例：

```text
file://docs/FastAPI.md
db://orders/schema
obsidian://python/Agent/03.LangChain
```

适合：

- 文档内容
- 数据库 schema
- 项目文件
- 配置说明
- 日志片段

### 3.3 Prompts

Prompts 是可复用提示词模板。

示例：

```text
code_review_prompt
rag_answer_prompt
sql_analysis_prompt
agent_planning_prompt
```

适合：

- 固定工作流
- 团队统一提示词
- 复杂任务模板
- 带参数的 Prompt

## 4. MCP 和 LangChain Tool 的区别

LangChain Tool 是框架内部的工具抽象。

MCP 是跨客户端、跨应用的工具协议。

| 对比项 | LangChain Tool | MCP |
| --- | --- | --- |
| 定位 | LangChain 内部工具 | 通用上下文和工具协议 |
| 使用范围 | LangChain Agent | 支持 MCP 的客户端都可用 |
| 复用性 | 主要在项目内复用 | 可跨应用复用 |
| 能力类型 | 主要是 tool | tools、resources、prompts |
| 适合场景 | 快速开发 Agent | 标准化工具服务 |

可以组合使用：

```text
业务 API -> MCP Server -> MCP Client -> LangChain Tool -> Agent
```

## 5. 传输方式

MCP 使用 JSON-RPC 消息通信。

当前常见标准传输：

| 传输方式 | 适用场景 |
| --- | --- |
| stdio | 本地工具、命令行工具、IDE 插件 |
| Streamable HTTP | 远程 MCP Server、Web 服务、多客户端连接 |

### 5.1 stdio

stdio 模式下，客户端启动 MCP Server 子进程，通过标准输入输出通信。

```text
MCP Client -> 启动本地进程 -> stdin/stdout 交换 JSON-RPC 消息
```

适合：

- 本地文件工具
- 本地 Git 工具
- 本地 Obsidian 笔记工具
- 命令行脚本能力

注意：

- stdout 只能输出 MCP 协议消息。
- 日志应该写 stderr。
- 不要在 stdout 打印普通 debug 文本。

### 5.2 Streamable HTTP

Streamable HTTP 适合远程服务。

```text
MCP Client -> HTTP -> MCP Server
```

适合：

- 公司内部工具服务
- 云端业务 API 封装
- 多用户访问
- 需要鉴权和网络部署的场景

## 6. 适合封装成 MCP 的能力

本地能力：

- 读取 Obsidian 笔记
- 搜索项目文件
- 查询 Git 状态
- 读取日志文件
- 解析 PDF 或 Markdown

业务能力：

- 查询订单
- 查询用户信息
- 创建工单
- 查询库存
- 查询数据库
- 调用内部 HTTP API

AI 应用能力：

- RAG 检索
- 文档摘要
- Prompt 模板
- 数据分析工具
- 报告生成工具

## 7. 工具设计原则

MCP Tool 和 Agent Tool 一样，需要强约束。

| 原则 | 说明 |
| --- | --- |
| 小而明确 | 一个工具只做一件事 |
| 参数结构化 | 不要只传一个 input 字符串 |
| 描述具体 | 工具说明要写清楚使用场景 |
| 返回稳定 | 返回结构固定，便于模型理解 |
| 错误可读 | 不要只返回底层异常 |
| 权限内置 | 工具内部做权限判断 |
| 高风险确认 | 写操作不要默认直接执行 |

tools：

- `read_note`：读取指定笔记内容
- `search_notes`：按关键词搜索笔记

- `obsidian://python/Agent`
- `obsidian://python/Agent/03.LangChain`

prompts：

- `summarize_note_prompt`
- `generate_review_questions_prompt`
