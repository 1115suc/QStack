## 项目一：企业知识库 RAG 问答系统

### 1. 项目背景

企业内部文档分散在 Markdown、PDF、接口文档、业务手册中，人工搜索效率低。系统通过 RAG 将文档转成可检索知识库，用户可以自然语言提问，并获得带引用来源的答案。

### 2. 核心功能

- 文档上传
- 文档解析
- 文本分块
- 向量化入库
- 知识库问答
- 引用来源返回
- 检索日志
- 问答历史

### 3. 技术栈

- FastAPI
- LangChain
- Chroma / Qdrant / Milvus
- Redis
- Docker
- LangSmith / Langfuse

### 4. 可扩展亮点

- 混合检索
- rerank
- 查询改写
- 多知识库隔离
- 文档增量更新
- RAG 自动化评估

### 5. 面试表达

> 这个项目不是简单把文档塞给大模型，而是完整实现了文档解析、分块、向量索引、检索、生成、引用来源和评估。我重点解决了召回不准、上下文过长、答案不可追溯的问题，并通过日志记录每次检索命中的 chunk 和相似度分数，方便排查错误回答。

## 项目二：多工具 AI 助手 Agent

### 1. 项目背景

业务人员经常需要查询订单、搜索知识库、调用外部接口、整理结果。系统通过 Agent 将多个工具统一编排，用户只需要用自然语言描述任务。

### 2. 核心功能

- 多轮对话
- 工具调用
- 订单查询
- 知识库检索
- HTTP API 调用
- 工具调用日志
- 失败重试
- 敏感操作人工确认

### 3. 技术栈

- FastAPI
- LangChain
- LangGraph
- Tool Calling
- Redis
- MCP
- Docker Compose

### 4. 可扩展亮点

- LangGraph 状态流
- MCP 工具服务
- 人工确认节点
- trace 回放
- token 成本统计

### 5. 面试表达

> 我把 Agent 当成一个有外部依赖的后端系统来设计，而不是只跑框架示例。每个工具都有 Schema、权限校验、超时和错误处理；每次执行都有 request_id，可以追踪模型输入、工具调用、耗时和最终输出。对于写操作，我加入了人工确认，避免模型误调用造成业务风险。

## 简历写法模板

```text
企业知识库 RAG 问答系统

技术栈：FastAPI、LangChain、Qdrant、Redis、Docker、LangSmith

项目描述：
基于企业内部文档构建知识库问答系统，支持文档上传、向量索引、语义检索、引用来源返回和问答日志追踪。

核心工作：
1. 使用 RecursiveCharacterTextSplitter 对文档进行分块，并保留文件名、页码、标题等 metadata。
2. 基于向量检索 + rerank 优化召回质量，降低无关上下文进入 Prompt 的概率。
3. 使用 FastAPI 封装文档上传、索引构建、问答接口，并通过 Redis 缓存高频问题。
4. 接入 LangSmith 追踪模型调用、检索结果、token 消耗和接口耗时。
```
