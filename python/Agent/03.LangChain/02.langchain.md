# LangChain进阶篇

## 1. 进阶开发的核心变化

基础阶段关注模型是否能回答；进阶阶段关注系统是否可控：

- 输出要能被程序消费，而不是只返回自然语言。
- 工具调用要有 schema、权限、超时、重试和审计。
- 对话要有 session 级状态。
- RAG 要能引用来源，避免只“看起来像正确”。
- 流式输出要区分 token、步骤、工具进度。
- Trace 要能支持排障、回放和评估。

## 2. 结构化输出

当下游要读取模型结果时，优先使用结构化输出，而不是让模型“按 JSON 格式回答”。

```python
from pydantic import BaseModel, Field
from langchain.agents import create_agent

class TicketTriage(BaseModel):
    category: str = Field(description="问题类别，例如 billing、bug、feature")
    priority: str = Field(description="优先级：low、medium、high")
    summary: str = Field(description="一句话摘要")

agent = create_agent(
    model="openai:gpt-4.1-mini",
    tools=[],
    response_format=TicketTriage,
    system_prompt="你负责客服工单分诊，只输出结构化结果。",
)

result = agent.invoke(
    {
        "messages": [
            {"role": "user", "content": "用户说账单被扣了两次，希望马上处理。"}
        ]
    }
)

triage: TicketTriage = result["structured_response"]
print(triage.category, triage.priority, triage.summary)
```

实践建议：

- 用 Pydantic 表达字段、类型、说明和约束。
- 对枚举值使用 `Literal` 或 Enum。
- 对生产接口返回结构化对象，不返回模型原文。
- 保存原始消息和结构化结果，便于审计。

## 3. 工具设计进阶

工具质量直接决定 Agent 上限。工具不是简单函数暴露，而是受控 API adapter。

```python
from typing import Literal
from pydantic import BaseModel, Field
from langchain_core.tools import tool

class RefundInput(BaseModel):
    order_id: str = Field(description="订单编号")
    reason: str = Field(description="退款原因")
    amount_cents: int = Field(ge=1, description="退款金额，单位为分")
    mode: Literal["dry_run", "commit"] = Field(
        default="dry_run",
        description="dry_run 只校验，commit 才真正执行",
    )

@tool(args_schema=RefundInput)
def request_refund(
    order_id: str,
    reason: str,
    amount_cents: int,
    mode: str = "dry_run",
) -> str:
    """校验或发起退款。高风险操作默认 dry_run。"""
    if mode != "commit":
        return f"退款校验通过：{order_id} 可退 {amount_cents} 分。"
    return f"退款已提交：{order_id}，金额 {amount_cents} 分。"
```

工具设计原则：

- **幂等优先**：查询类工具最安全；写操作必须有业务幂等键。
- **高风险操作默认 dry-run**：让模型先解释计划，再由用户或策略确认。
- **最小权限**：工具内部按用户身份和 scope 校验权限。
- **强 schema**：字段名和描述要让模型知道何时调用。
- **返回短而结构化**：不要把几百 KB 原始 JSON 直接塞回上下文。

## 4. Runtime Context

不要把用户 ID、租户 ID、权限、token 直接写进 prompt。应通过 runtime context 传给工具或 middleware。

```python
from dataclasses import dataclass
from langchain.agents import create_agent
from langchain_core.tools import tool
from langchain_core.runnables import RunnableConfig

@dataclass
class UserContext:
    user_id: str
    tenant_id: str
    scopes: list[str]

@tool
def list_user_orders(config: RunnableConfig) -> str:
    """查询当前用户订单。"""
    ctx: UserContext = config["configurable"]["user_context"]
    return f"tenant={ctx.tenant_id}, user={ctx.user_id} 的订单列表"

agent = create_agent(
    model="openai:gpt-4.1-mini",
    tools=[list_user_orders],
)

result = agent.invoke(
    {"messages": [{"role": "user", "content": "查我的订单"}]},
    config={
        "configurable": {
            "user_context": UserContext(
                user_id="u_123",
                tenant_id="t_001",
                scopes=["orders:read"],
            )
        }
    },
)
```

上下文分层建议：

| 类型 | 放哪里 |
| --- | --- |
| 用户自然语言诉求 | `messages` |
| 系统行为规则 | `system_prompt` |
| 用户身份、租户、权限 | runtime config / context |
| API key、数据库凭据 | 服务端环境变量或 secret manager |
| 可回放状态 | checkpoint / state store |

## 5. 短期记忆与会话状态

LangChain agent 底层基于 LangGraph。多轮会话应使用 thread / checkpoint 思路，而不是手动把所有历史塞进 prompt。

典型调用形态：

```python
config = {
    "configurable": {
        "thread_id": "session-user-123"
    }
}

agent.invoke(
    {"messages": [{"role": "user", "content": "我叫小王"}]},
    config=config,
)

agent.invoke(
    {"messages": [{"role": "user", "content": "我叫什么？"}]},
    config=config,
)
```

实践建议：

- `thread_id` 对应业务 session，不要随请求随机生成。
- 控制历史长度，长会话需要摘要或持久化记忆。
- 不要把敏感凭据写入消息历史。
- 对企业场景，checkpoint 存储要区分租户。

## 6. RAG 基础架构

RAG 不是“向量搜索 + prompt”这么简单。一个可用链路通常包含：

1. 文档解析：PDF、HTML、Markdown、Office 文档。
2. 清洗切块：按标题、段落、语义边界切分。
3. Embedding：为 chunk 生成向量。
4. Vector Store：Milvus、Qdrant、Chroma、PGVector 等。
5. 检索：相似度、metadata filter、hybrid search。
6. Rerank：用 reranker 重新排序候选 chunk。
7. 生成：带引用回答。
8. 评估：faithfulness、answer correctness、context precision。

最小 RAG 示例：

```python
from langchain.chat_models import init_chat_model
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_messages(
    [
        (
            "system",
            "你只能基于给定上下文回答。若上下文不足，回答：资料不足。\n\n上下文：\n{context}",
        ),
        ("human", "{question}"),
    ]
)

model = init_chat_model("openai:gpt-4.1-mini", temperature=0)

rag_chain = prompt | model | StrOutputParser()

answer = rag_chain.invoke(
    {
        "question": "LangChain 的 agent 底层依赖什么？",
        "context": "LangChain agents are built on top of LangGraph ...",
    }
)
```

真实项目中，`context` 来自 retriever，而不是手写字符串。

## 7. Retriever 工具化

Agentic RAG 常把检索器包装成工具，让模型决定何时检索：

```python
from langchain_core.tools import tool

@tool
def search_internal_docs(query: str) -> str:
    """检索内部知识库。适合回答产品、流程、技术文档问题。"""
    docs = retriever.invoke(query)
    return "\n\n".join(
        f"[{idx}] {doc.page_content}\nsource={doc.metadata.get('source')}"
        for idx, doc in enumerate(docs, start=1)
    )
```

这种方式适合：

- 用户问题并不总需要知识库。
- Agent 有多个数据源，需要自己选择。
- 需要先问清楚，再检索。

不适合：

- 强监管问答，必须每次检索并引用。
- 低延迟问答，Agent 多步推理开销过大。
- 结果必须完全可预测的检索问答。

## 8. 引用与可追溯回答

RAG 的回答应保留引用：

```python
from pydantic import BaseModel, Field

class CitedAnswer(BaseModel):
    answer: str = Field(description="基于上下文的回答")
    citations: list[str] = Field(description="引用来源 ID 或 URL")
    insufficient_context: bool = Field(description="上下文是否不足")
```

生成规则建议写入 system prompt：

```text
你只能使用给定 context。
每个关键结论都必须给出 citations。
如果 context 中没有答案，insufficient_context=true。
不要编造来源。
```

工程上要保存：

- query
- retrieved chunk IDs
- reranked chunk IDs
- prompt context
- final answer
- citations
- trace URL

这样才能做离线评估和线上问题复盘。

## 9. 流式输出进阶

Agent 的流式输出可以用于不同 UI：

| stream mode | 适用场景 |
| --- | --- |
| `messages` | 聊天界面 token 流 |
| `updates` | 显示“正在检索 / 正在调用工具 / 正在总结” |
| `custom` | 工具内部自定义进度，例如已处理第几个文件 |

FastAPI SSE 示例骨架：

```python
import json
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

def sse_event(data: dict) -> str:
    return f"data: {json.dumps(data, ensure_ascii=False)}\n\n"

@app.get("/chat/stream")
def chat_stream(q: str):
    def gen():
        for chunk in agent.stream(
            {"messages": [{"role": "user", "content": q}]},
            stream_mode="updates",
        ):
            yield sse_event(chunk)

    return StreamingResponse(gen(), media_type="text/event-stream")
```

生产注意：

- SSE 连接要处理客户端断开。
- 长工具调用应输出心跳或进度。
- 前端不要假设每个 chunk 都是最终文本。
- 错误事件要结构化返回。

## 10. 错误处理与重试

常见错误来源：

- 模型限流或超时
- 工具 API 超时
- 工具参数不合法
- 结构化输出解析失败
- 检索无结果
- 上下文超长

处理策略：

| 错误    | 建议                         |
| ----- | -------------------------- |
| 模型限流  | 指数退避、provider fallback、排队  |
| 工具超时  | 设置 timeout，返回可解释错误         |
| 参数不合法 | schema 校验 + 让模型修正参数        |
| 解析失败  | 使用结构化输出能力，不靠手写 JSON prompt |
| 检索无结果 | 明确回答资料不足                   |
| 上下文超长 | rerank、摘要、压缩、限制 chunk 数    |

不要让工具异常直接冒泡成 500 给用户。Agent 工具应返回模型能理解的错误摘要，同时服务端记录完整异常。

## 11. FastAPI 工程结构建议

```text
app/
  main.py
  api/
    chat.py
  agents/
    assistant.py
    prompts.py
    tools.py
    schemas.py
  rag/
    loaders.py
    splitters.py
    retrievers.py
    citations.py
  infra/
    settings.py
    tracing.py
    auth.py
    rate_limit.py
  evals/
    datasets/
    test_rag_quality.py
```

建议：

- `agents/assistant.py` 只负责组装 Agent。
- 工具函数不要直接写在路由里。
- prompt 单独管理，便于版本化。
- RAG 索引构建和在线查询分离。
- trace、auth、rate limit 属于基础设施层。
