# AI 应用部署与监控

> 目标：把 AI 应用从本地 Demo 部署成可观测、可排查、可优化的服务。

---

## 一、为什么 AI 应用更需要监控

传统后端关注：

- QPS
- 延迟
- 错误率
- CPU / 内存

AI 应用还要关注：

- token 消耗
- 模型调用耗时
- 工具调用耗时
- 检索命中率
- 上下文长度
- 单次请求成本
- 幻觉和错误回答
- Agent 工具调用链路

## 二、部署推荐结构

```text
FastAPI API
  |
  |-- LLM Provider
  |-- Vector DB
  |-- Redis
  |-- PostgreSQL
  |-- LangSmith / Langfuse
  |-- Prometheus
  |-- Grafana
```

本地学习阶段可以用 Docker Compose 编排。

## 三、Dockerfile 示例

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

## 四、docker-compose 示例

```yaml
services:
  api:
    build: .
    ports:
      - "8000:8000"
    env_file:
      - .env
    depends_on:
      - redis

  redis:
    image: redis:7
    ports:
      - "6379:6379"
```

## 五、必须记录的指标

| 指标 | 作用 |
| --- | --- |
| request_count | 请求量 |
| request_latency_ms | 接口延迟 |
| model_latency_ms | 模型调用耗时 |
| tool_latency_ms | 工具调用耗时 |
| prompt_tokens | 输入 token |
| completion_tokens | 输出 token |
| total_cost | 单次调用成本 |
| retrieval_hit_rate | 检索命中率 |
| error_count | 错误数量 |

## 六、日志建议

日志尽量结构化：

```json
{
  "request_id": "req_001",
  "user_id": "u_001",
  "route": "/agent/chat",
  "model": "gpt-4.1-mini",
  "latency_ms": 1850,
  "prompt_tokens": 1200,
  "completion_tokens": 180,
  "tool_calls": ["search_docs"],
  "status": "success"
}
```

## 七、上线 Checklist

- `.env` 不提交到 Git
- API key 不写死在代码里
- 外部请求有 timeout
- 工具有权限校验
- 日志敏感信息脱敏
- 请求有 request_id
- 有健康检查接口 `/health`
- 有 Dockerfile
- 有 README 部署说明
- 有错误率和耗时监控
