# LangSmith监控工具

## 1. 为什么学习 LangSmith

LangSmith 是 LangChain 官方生态中的 LLM 应用可观测、调试和评估平台。

如果你的主线是：

- LangChain
- LangGraph
- RAG
- Agent
- Tool Calling

那么 LangSmith 非常值得学习，因为它和 LangChain / LangGraph 集成最自然。

LangSmith 主要解决这些问题：

- 一次 Agent 请求中模型调用了什么？
- RAG 检索到了哪些上下文？
- 工具调用参数和结果是什么？
- LangGraph 每个节点如何流转？
- Prompt 修改后效果是否变好？
- 如何用数据集批量测试 RAG / Agent？
- 线上失败样本如何沉淀成评估集？

可以把 LangSmith 理解为：

```text
LangChain / LangGraph 应用的 Trace + Debug + Evaluation 平台
```

## 2. LangSmith 适合监控什么

| 监控对象 | 说明 |
| --- | --- |
| Chain | Prompt、模型调用、输出解析 |
| RAG | 检索 query、召回文档、最终答案 |
| Agent | 工具选择、工具调用、最终回答 |
| LangGraph | 节点输入输出、分支、状态流转 |
| LLM 调用 | 模型、输入、输出、token、耗时 |
| 数据集 | 测试样本、参考答案、版本 |
| 评估实验 | 不同 Prompt、模型、参数的效果对比 |

LangSmith 不只是日志工具，更重要的是评估闭环。

```text
线上 Trace -> 发现问题 -> 加入 Dataset -> 跑 Evaluation -> 优化 Prompt / RAG / Agent
```

## 3. 核心概念

| 概念 | 说明 |
| --- | --- |
| Run | 一次执行记录，可以是 LLM、Chain、Tool、Retriever |
| Trace | 一次完整请求的调用树 |
| Project | 一组 traces 的归类 |
| Dataset | 评估数据集 |
| Example | Dataset 中的一条测试样本 |
| Experiment | 在数据集上运行的一次评估实验 |
| Evaluator | 评分器，可以是规则、人工、LLM-as-a-Judge |
| Playground | 在 UI 中调试 Prompt、模型和参数 |

一次请求可以拆成树状结构：

```text
Trace: agent_chat
  Run: classify_question
  Run: retrieve_documents
  Run: llm_decide_tool
  Run: tool:get_order_status
  Run: final_answer
```

## 4. 安装和环境变量

安装：

```bash
pip install -U langsmith
```

如果使用 LangChain / LangGraph：

```bash
pip install -U langchain langgraph langchain-openai
```

配置环境变量：

```powershell
$env:LANGSMITH_TRACING="true"
$env:LANGSMITH_API_KEY="lsv2_..."
$env:LANGSMITH_PROJECT="ai-agent-dev"
```

如果使用旧配置，也可能看到：

```powershell
$env:LANGCHAIN_TRACING_V2="true"
$env:LANGCHAIN_API_KEY="lsv2_..."
$env:LANGCHAIN_PROJECT="ai-agent-dev"
```

实际项目中建议统一使用 LangSmith 新命名，并把配置写入 `.env`。

## 5. LangChain 自动 tracing

LangChain 应用开启 tracing 后，很多调用会自动进入 LangSmith。

```python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_template(
    "请用一句话解释：{question}"
)

llm = ChatOpenAI(model="gpt-4.1-mini", temperature=0)

chain = prompt | llm

result = chain.invoke(
    {"question": "什么是 RAG？"},
    config={
        "metadata": {
            "module": "rag",
            "env": "dev",
        },
        "tags": ["demo", "rag"],
    },
)

print(result.content)
```

在 LangSmith 中可以看到：

- prompt 输入
- model 名称
- 模型输出
- token 使用量
- 耗时
- metadata
- tags

## 6. 自定义函数 tracing

如果某些逻辑不是 LangChain 组件，可以使用 `traceable` 装饰器。

```python
from langsmith import traceable

@traceable(name="retrieve_documents")
def retrieve_documents(query: str, top_k: int = 4):
    docs = vector_store.similarity_search(query, k=top_k)
    return [
        {
            "source": doc.metadata.get("source"),
            "content": doc.page_content[:300],
        }
        for doc in docs
    ]
```

调用：

```python
docs = retrieve_documents("FastAPI Depends 有什么用？", top_k=4)
```

适合记录：

- RAG 检索
- rerank
- 工具调用
- HTTP API 请求
- LangGraph 节点
- 数据库查询

## 7. RAG 中如何使用 LangSmith

RAG 需要重点记录检索过程。

建议记录：

```json
{
  "question": "FastAPI Depends 有什么用？",
  "rewritten_query": "FastAPI Depends 依赖注入作用",
  "retrieved_docs": [
    {
      "source": "FastAPI.md",
      "chunk_id": "fastapi_001",
      "score": 0.83
    }
  ],
  "answer": "Depends 用于声明依赖注入..."
}
```

RAG 排查方式：

| 问题 | 看什么 |
| --- | --- |
| 答案错误 | 检索文档是否正确 |
| 答案幻觉 | Prompt 是否限制只基于上下文 |
| 引用错误 | metadata 是否正确 |
| 召回差 | query rewrite、top_k、embedding |
| 成本高 | context 长度、prompt_tokens |

RAG trace 推荐结构：

```text
Trace: rag_chat
  Run: rewrite_query
  Run: retrieve_documents
  Run: rerank_documents
  Run: generate_answer
```

## 8. Agent 中如何使用 LangSmith

Agent 最容易出问题的地方是工具选择和工具参数。

Agent trace 需要关注：

- 模型为什么选择某个工具
- 工具参数是否正确
- 工具返回是否符合预期
- 是否重复调用工具
- 是否达到最大步骤数
- 最终答案是否基于工具结果

工具调用记录示例：

```json
{
  "tool": "get_order_status",
  "args": {
    "order_id": "A1001"
  },
  "output": {
    "status": "已支付，等待发货"
  },
  "latency_ms": 42
}
```

排查 Agent 问题：

| 问题 | 排查方向 |
| --- | --- |
| 调错工具 | 工具描述、工具列表、Prompt |
| 参数错误 | Tool Schema、用户上下文 |
| 死循环 | 步骤数、条件路由、终止条件 |
| 越权调用 | 工具过滤、runtime context |
| 回答不准 | 工具结果、最终生成 Prompt |

## 9. LangGraph 中如何使用 LangSmith

LangGraph 的优势是状态图，但生产中必须能看清每个节点。

建议每个节点都能在 trace 中体现：

```text
Trace: langgraph_agent
  Node: classify
  Node: retrieve
  Node: tool_call
  Node: approval
  Node: generate
```

每个节点建议记录：

- node name
- input state 摘要
- output state 摘要
- route decision
- latency
- error

示例 metadata：

```python
result = graph_app.invoke(
    {"question": "帮我查订单 A1001"},
    config={
        "metadata": {
            "thread_id": "s_001",
            "user_id": "u_001",
        },
        "tags": ["langgraph", "agent"],
    },
)
```

## 10. Dataset 数据集

LangSmith 的评估依赖 Dataset。

一个 Dataset 可以保存：

- 输入问题
- 参考答案
- 期望工具调用
- 期望引用来源
- 分类标签

示例：

```json
{
  "inputs": {
    "question": "FastAPI Depends 的作用是什么？"
  },
  "outputs": {
    "answer": "Depends 用于声明依赖注入，常用于鉴权、数据库连接、公共参数等。"
  },
  "metadata": {
    "tags": ["FastAPI", "依赖注入"],
    "type": "rag"
  }
}
```

Dataset 来源：

- 手工整理的测试题
- 线上失败 trace
- 用户高频问题
- 面试题库
- RAG 错误案例

## 11. Evaluation 评估

LangSmith Evaluation 用于比较不同应用配置的质量。

可以评估：

- 不同 Prompt
- 不同模型
- 不同 temperature
- 不同 chunk_size
- 不同 retriever
- 是否使用 rerank

常见评估方式：

| 评估方式 | 说明 |
| --- | --- |
| 规则评估 | 字符串匹配、JSON 字段检查 |
| 人工评估 | 人工标注好坏 |
| LLM-as-a-Judge | 用模型评分 |
| 对比评估 | 比较两个实验哪个更好 |

RAG 评估项：

- 是否命中正确来源
- 答案是否忠于上下文
- 答案是否完整
- 是否有幻觉

Agent 评估项：

- 是否调用正确工具
- 工具参数是否正确
- 是否完成任务
- 是否避免高风险误操作

## 12. evaluate 简单示例

伪代码结构：

```python
from langsmith import Client
from langsmith.evaluation import evaluate

client = Client()

def target_app(inputs: dict):
    question = inputs["question"]
    answer = rag_service.chat(question)
    return {"answer": answer}

def exact_match(outputs: dict, reference_outputs: dict):
    return outputs["answer"] == reference_outputs["answer"]

results = evaluate(
    target_app,
    data="fastapi-rag-dataset",
    evaluators=[exact_match],
    experiment_prefix="rag-baseline",
)
```

实际项目中，评估器可以更复杂，例如：

- 判断答案是否包含关键点
- 判断引用来源是否正确
- 使用 LLM 对 correctness 打分
- 比较两个 Prompt 的输出质量

## 13. Playground

LangSmith Playground 适合做 Prompt 和模型调试。

可以在 Playground 中：

- 修改 Prompt
- 切换模型
- 调整 temperature
- 对同一数据集批量运行
- 比较不同版本输出
- 把效果好的配置沉淀下来

适合流程：

```text
发现坏 trace -> 打开 Playground -> 修改 Prompt -> 在 Dataset 上评估 -> 通过后更新代码
```

不要只凭单个样本感觉改 Prompt，要用 Dataset 批量验证。

## 14. LangSmith 和 Langfuse 的区别

| 对比项 | LangSmith | Langfuse |
| --- | --- | --- |
| 生态 | LangChain 官方生态 | 通用 LLM 可观测平台 |
| LangChain 集成 | 非常自然 | 也支持，但不是官方核心 |
| LangGraph 支持 | 很适合 | 也可记录 |
| 开源/自托管 | 以平台服务为主 | 开源和自托管更常见 |
| 重点 | tracing、debug、eval、dataset | tracing、prompt、analytics、eval |
| 推荐场景 | LangChain/LangGraph 项目 | 多框架、想自托管、统一观测 |

学习建议：

- 如果你主学 LangChain / LangGraph，先学 LangSmith。
- 如果你想做开源自托管观测平台，再学 Langfuse。
- 面试中可以说：LangSmith 更适合 LangChain 生态调试评估，Langfuse 更适合通用 LLM 观测和自托管。

## 15. 生产级 Checklist

- 是否开启 tracing
- 是否设置 project
- 是否用 tags 区分 RAG、Agent、LangGraph
- 是否记录 user_id 和 session_id
- 是否记录 RAG 检索结果
- 是否记录工具调用参数
- 是否记录 token 和 latency
- 是否从失败 trace 建 Dataset
- 是否有离线评估集
- Prompt 修改是否跑 evaluation
- 是否比较不同模型和 retriever
- 是否定期复盘线上失败样本

## 16. 实践任务

给你的 AI 应用接入 LangSmith：

- 配置 `LANGSMITH_TRACING`
- 配置 `LANGSMITH_API_KEY`
- 配置 `LANGSMITH_PROJECT`
- 在 RAG 服务中记录 query、docs、answer
- 在 Agent 服务中记录 tool call
- 在 LangGraph 中记录 node 和 route
- 建一个 30 条样本的 RAG Dataset
- 对两个 Prompt 版本跑 evaluation
- 把失败样本加入 Dataset

目标：让你的 AI 应用具备“可调试、可评估、可迭代”的工程闭环。
