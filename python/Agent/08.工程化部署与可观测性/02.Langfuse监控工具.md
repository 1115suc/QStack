# Langfuse监控工具

## 1. 为什么推荐 Langfuse

AI 应用开发中，传统后端监控只能看到接口是否成功、耗时多少、错误率多少。

但 RAG 和 Agent 还需要回答这些问题：

- 模型到底收到了什么 Prompt？
- 检索到了哪些文档？
- 工具调用了几次？
- 哪一步最慢？
- 单次请求花了多少 token？
- 为什么 Agent 调错工具？
- RAG 错误是检索错了，还是模型生成错了？
- Prompt 修改后效果有没有变好？

Langfuse 是一个面向 LLM 应用的可观测性平台，适合用来监控 RAG、Agent、LangChain、LangGraph 这类应用。

它的核心价值是：

```text
把一次 AI 请求拆成可追踪、可分析、可评估的 Trace
```

## 2. Langfuse 适合监控什么

| 监控对象 | 说明 |
| --- | --- |
| LLM 调用 | 模型、Prompt、输出、token、耗时、成本 |
| RAG 检索 | 用户问题、检索 query、召回文档、分数、来源 |
| Agent 工具调用 | 工具名、参数、结果、耗时、错误 |
| LangGraph 节点 | 每个节点输入、输出、分支、耗时 |
| 用户会话 | 同一个 session 下的多轮对话 |
| Prompt 版本 | Prompt 的版本、修改、回滚 |
| 评估结果 | 人工评分、LLM-as-a-Judge、自动评估 |

如果你做的是 AI 应用开发工程师路线，Langfuse 比单纯学 Prometheus 更贴近岗位需求。

Prometheus / Grafana 仍然有价值，但它们更适合监控：

- CPU
- 内存
- QPS
- 接口错误率
- 服务可用性

Langfuse 更适合监控：

- Prompt
- token
- tool call
- RAG context
- Agent trace
- LLM 质量

## 3. Langfuse 核心概念

| 概念 | 说明 |
| --- | --- |
| Trace | 一次完整 AI 请求的链路 |
| Observation | Trace 中的一个步骤，例如 LLM 调用、工具调用、检索 |
| Generation | 一次模型生成 |
| Span | 一个普通执行步骤，例如 RAG 检索、工具执行 |
| Session | 多轮对话会话 |
| User | 用户维度分析 |
| Score | 人工或自动评分 |
| Dataset | 测试集和评估样本 |
| Prompt Management | Prompt 版本管理 |

可以这样理解：

```text
Trace = 一次完整请求
Observation = 请求中的每个步骤
Generation = 模型调用
Span = 工具、检索、业务逻辑
Score = 对结果质量的评价
```

## 4. RAG 中如何使用 Langfuse

RAG 请求建议记录这些信息：

```json
{
  "trace_name": "rag_chat",
  "user_id": "u_001",
  "session_id": "s_001",
  "input": "FastAPI Depends 有什么用？",
  "retrieval": {
    "query": "FastAPI Depends 依赖注入作用",
    "top_k": 4,
    "documents": [
      {
        "source": "FastAPI.md",
        "chunk_id": "fastapi_001",
        "score": 0.83
      }
    ]
  },
  "generation": {
    "model": "gpt-4.1-mini",
    "prompt_tokens": 1200,
    "completion_tokens": 180
  }
}
```

排查 RAG 问题时，优先看：

| 问题 | 在 Langfuse 中看什么 |
| --- | --- |
| 答案不相关 | 检索 query、召回文档 |
| 答案编造 | Prompt、context、模型输出 |
| 回答太慢 | 检索耗时、模型耗时 |
| 成本太高 | prompt_tokens、completion_tokens |
| 引用错误 | source metadata、chunk 内容 |

## 5. Agent 中如何使用 Langfuse

Agent 请求建议记录：

- 用户输入
- 使用的模型
- 工具列表
- 每次工具调用
- 工具参数
- 工具结果
- 工具耗时
- 最终答案
- token 成本

示例 trace：

```text
Trace: agent_chat
  Span: classify_question
  Generation: llm_decide_tool
  Span: tool:get_order_status
  Generation: final_answer
```

工具调用日志示例：

```json
{
  "tool": "get_order_status",
  "args": {
    "order_id": "A1001"
  },
  "result": {
    "status": "已支付，等待发货"
  },
  "latency_ms": 42,
  "status": "success"
}
```

排查 Agent 问题时，重点看：

| 问题 | 排查方向 |
| --- | --- |
| 调错工具 | 工具描述、工具列表、模型选择过程 |
| 参数填错 | Tool Schema、用户上下文 |
| 死循环 | 工具调用次数、步骤数 |
| 写操作风险 | 是否经过 approval / dry-run |
| 回答错误 | 工具结果是否正确、最终 Prompt 是否清晰 |

## 6. 安装 Python SDK

```bash
pip install langfuse
```

配置环境变量：

```powershell
$env:LANGFUSE_PUBLIC_KEY="pk-lf-..."
$env:LANGFUSE_SECRET_KEY="sk-lf-..."
$env:LANGFUSE_HOST="https://cloud.langfuse.com"
```

如果是自托管，把 `LANGFUSE_HOST` 改成自己的服务地址。

```powershell
$env:LANGFUSE_HOST="http://localhost:3000"
```

## 7. 最小使用示例

```python
from langfuse import Langfuse

langfuse = Langfuse()

trace = langfuse.trace(
    name="rag_chat",
    user_id="u_001",
    session_id="s_001",
    input="FastAPI Depends 有什么用？",
)

retrieval_span = trace.span(
    name="retrieve_documents",
    input={"query": "FastAPI Depends 依赖注入作用", "top_k": 4},
)

retrieval_span.end(
    output={
        "documents": [
            {
                "source": "FastAPI.md",
                "chunk_id": "fastapi_001",
                "score": 0.83,
            }
        ]
    }
)

generation = trace.generation(
    name="generate_answer",
    model="gpt-4.1-mini",
    input=[
        {"role": "system", "content": "你是一个严谨的知识库问答助手。"},
        {"role": "user", "content": "FastAPI Depends 有什么用？"},
    ],
)

generation.end(
    output="Depends 用于声明依赖注入，常用于鉴权、数据库连接、公共参数等场景。",
    usage={
        "input": 1200,
        "output": 180,
    },
)

trace.update(
    output="Depends 用于声明依赖注入，常用于鉴权、数据库连接、公共参数等场景。"
)

langfuse.flush()
```

实际项目中，可以把这些逻辑封装到 `tracing/langfuse_client.py`。

## 8. LangChain 接入思路

Langfuse 可以和 LangChain / LangGraph 结合使用。

在项目中推荐做法：

```text
FastAPI 请求进来
  -> 创建 trace
  -> RAG 检索记录 span
  -> LLM 调用记录 generation
  -> Tool 调用记录 span
  -> 返回答案并更新 trace output
```

项目结构：

```text
app/
  tracing/
    langfuse_client.py
  services/
    rag_service.py
    agent_service.py
  tools/
    order_tool.py
```

封装示例：

```python
from langfuse import Langfuse

class LangfuseTracer:
    def __init__(self):
        self.client = Langfuse()

    def create_trace(self, name: str, user_id: str, session_id: str, input_text: str):
        return self.client.trace(
            name=name,
            user_id=user_id,
            session_id=session_id,
            input=input_text,
        )

    def flush(self):
        self.client.flush()
```

## 10. 评分和评估

Langfuse 不只是看日志，还可以做评分。

常见评分：

| 评分项 | 说明 |
| --- | --- |
| correctness | 答案是否正确 |
| relevance | 答案是否相关 |
| faithfulness | 是否忠于上下文 |
| helpfulness | 是否有帮助 |
| user_feedback | 用户点赞或点踩 |

RAG 推荐评分：

- 检索结果是否命中正确资料
- 回答是否引用了正确来源
- 答案是否基于上下文

Agent 推荐评分：

- 是否调用了正确工具
- 工具参数是否正确
- 是否完成用户任务
- 是否避免了高风险误操作

## 11. 和 Prometheus / Grafana 的关系

Langfuse 不是完全替代 Prometheus / Grafana。

推荐组合：

```text
Langfuse：监控 AI 链路质量、Prompt、Trace、Token、Tool Call
Prometheus：监控服务指标、CPU、内存、QPS、错误率
Grafana：展示基础设施和服务指标看板
```

学习阶段可以先只用 Langfuse。

真实生产环境可以组合使用：

```text
FastAPI + Langfuse + Prometheus + Grafana
```

## 12. 生产级 Checklist

- 是否记录每次请求的 trace
- 是否记录 LLM 输入和输出
- 是否记录 token 和成本
- 是否记录 RAG 检索结果
- 是否记录工具调用参数
- 是否记录工具调用耗时
- 是否记录错误和异常
- 是否用 session_id 关联多轮对话
- 是否按 user_id 分析问题
- 是否支持用户反馈
- 是否建立评估集
- 是否定期查看失败 trace

## 13. 实践任务

给你的 RAG / Agent 服务接入 Langfuse：

- 安装 `langfuse`
- 配置 `LANGFUSE_PUBLIC_KEY`
- 配置 `LANGFUSE_SECRET_KEY`
- 配置 `LANGFUSE_HOST`
- 在 `/rag/chat` 中记录 RAG trace
- 在 `/agent/chat` 中记录工具调用 trace
- 记录 prompt_tokens、completion_tokens
- 记录检索 source 和 score
- 记录 tool name、args、latency
- 每周整理 10 条失败 trace 做优化

目标：以后面试时能讲清楚“我不只是做了 AI Demo，还能监控它为什么错、哪里慢、花多少钱、如何优化”。
