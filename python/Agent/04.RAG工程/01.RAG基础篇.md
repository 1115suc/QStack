# RAG基础篇

## 1. RAG 解决什么问题

RAG 是 Retrieval-Augmented Generation，中文通常叫检索增强生成。

大模型本身很强，但直接用于业务问答时会遇到几个典型问题：

- **不知道私有知识**：企业文档、业务手册、内部 API 文档不在模型训练数据里。
- **知识可能过期**：模型参数里的知识无法实时更新。
- **容易幻觉**：没有依据时也可能编造看似合理的答案。
- **上下文窗口有限**：不能把全部文档一次性塞进 prompt。
- **答案不可追溯**：用户不知道答案来自哪份资料。

RAG 的核心思路是：先从知识库检索相关资料，再让大模型基于检索结果回答。

```text
用户问题 -> 检索相关文档 -> 构造上下文 -> LLM 生成答案 -> 返回答案和引用来源
```

RAG 更适合应用开发工程师，因为它不要求训练模型，而是要求把文档、检索、模型和后端服务组织成一个可用系统。

## 2. RAG 基本架构

一个基础 RAG 系统一般分成两个阶段。

### 2.1 索引阶段

索引阶段负责把文档变成可检索的数据。

```text
原始文档 -> 文档加载 -> 文本清洗 -> 文本分块 -> Embedding -> 向量数据库
```

### 2.2 查询阶段

查询阶段负责根据用户问题找资料并生成答案。

```text
用户问题 -> 问题向量化 -> 向量检索 -> 组装 Prompt -> LLM 回答 -> 引用来源
```

常见模块：

| 模块 | 作用 |
| --- | --- |
| Document Loader | 加载 PDF、Markdown、网页、Word、CSV 等资料 |
| Text Splitter | 把长文本切成更适合检索的小块 |
| Embedding Model | 把文本转成向量 |
| Vector Store | 存储向量并支持相似度搜索 |
| Retriever | 根据问题取回相关文档块 |
| Prompt Template | 把问题和上下文组织成模型输入 |
| Chat Model | 根据上下文生成自然语言答案 |
| Source Metadata | 保存文件名、页码、标题等引用信息 |

## 4. Document 文档对象

LangChain 中常用 `Document` 表示一段文档内容。

```python
from langchain_core.documents import Document

doc = Document(
    page_content="FastAPI 是一个现代 Python Web 框架，适合构建高性能 API。",
    metadata={
        "source": "FastAPI.md",
        "page": 1,
        "title": "FastAPI 入门",
    },
)

print(doc.page_content)
print(doc.metadata["source"])
```

`page_content` 是文本内容，`metadata` 是附加信息。

RAG 项目中一定要重视 metadata，否则后续很难返回引用来源，也不方便排查错误答案。

## 5. 文档加载

不同文件类型需要不同 loader。

```python
from langchain_community.document_loaders import TextLoader

loader = TextLoader("docs/FastAPI.md", encoding="utf-8")
docs = loader.load()

print(len(docs))
print(docs[0].page_content[:100])
print(docs[0].metadata)
```

常见 loader：

| Loader | 适用场景 |
| --- | --- |
| `TextLoader` | 纯文本、Markdown |
| `PyPDFLoader` | PDF 文件 |
| `UnstructuredMarkdownLoader` | Markdown 结构化解析 |
| `CSVLoader` | CSV 数据 |
| `WebBaseLoader` | 网页内容 |
| `DirectoryLoader` | 批量加载目录 |

示例：批量加载 Markdown。

```python
from langchain_community.document_loaders import DirectoryLoader, TextLoader

loader = DirectoryLoader(
    path="docs",
    glob="**/*.md",
    loader_cls=TextLoader,
    loader_kwargs={"encoding": "utf-8"},
)

docs = loader.load()
print(f"加载文档数量：{len(docs)}")
```

## 6. 文本分块

为什么要分块？

模型不能一次性读取所有文档，向量检索也不适合直接检索整篇长文档。分块的目标是让每个 chunk 既足够短，又保留相对完整的语义。

常用分块器：

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=120,
    separators=["\n\n", "\n", "。", "，", " ", ""],
)

chunks = splitter.split_documents(docs)
print(f"切分后 chunk 数量：{len(chunks)}")
```

参数说明：

| 参数 | 说明 |
| --- | --- |
| `chunk_size` | 每个文本块的最大长度 |
| `chunk_overlap` | 相邻文本块之间的重叠长度 |
| `separators` | 优先按哪些分隔符切分 |

经验值：

| 文档类型 | chunk_size | chunk_overlap |
| --- | --- | --- |
| 普通说明文档 | 500 - 1000 | 80 - 150 |
| API 文档 | 800 - 1500 | 100 - 200 |
| FAQ | 按问答对切分 | 0 - 50 |
| 表格数据 | 按行或业务对象切分 | 视情况 |

## 7. Embedding 向量化

Embedding 的作用是把文本转成向量，让语义相近的文本在向量空间中距离更近。

```python
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small",
)

vector = embeddings.embed_query("FastAPI 适合做什么？")
print(len(vector))
```

选型建议：

- 中文知识库要选择中文效果稳定的 embedding。
- 业务文档比较短时，优先关注召回准确率。
- 文档量很大时，还要关注向量维度、成本和索引速度。
- embedding 模型更换后，通常需要重新构建索引。

## 8. 向量数据库 Chroma

Chroma 适合学习和本地 Demo。

```python
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

vector_store = Chroma.from_documents(
    documents=chunks,
    embedding=OpenAIEmbeddings(model="text-embedding-3-small"),
    persist_directory="./chroma_db",
)
```

加载已有索引：

```python
vector_store = Chroma(
    persist_directory="./chroma_db",
    embedding_function=OpenAIEmbeddings(model="text-embedding-3-small"),
)
```

常见向量库：

| 向量库 | 适用场景 |
| --- | --- |
| FAISS | 本地实验、内存索引 |
| Chroma | 学习、Demo、小型知识库 |
| Qdrant | 中小型生产项目，部署简单 |
| Milvus | 大规模向量检索 |
| Elasticsearch | 混合检索、关键词检索结合 |

## 9. 相似度检索

```python
query = "FastAPI 的优点是什么？"

docs = vector_store.similarity_search(query, k=3)

for doc in docs:
    print(doc.page_content[:120])
    print(doc.metadata)
```

带分数检索：

```python
results = vector_store.similarity_search_with_score(query, k=3)

for doc, score in results:
    print(score, doc.metadata.get("source"))
    print(doc.page_content[:120])
```

注意：不同向量库的 score 含义可能不同，有的是距离，有的是相似度，不能机械比较。

## 10. 最小 RAG Chain

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

prompt = ChatPromptTemplate.from_template(
    """
你是一个严谨的知识库问答助手。
请只根据上下文回答问题。
如果上下文中没有答案，请回答：资料中没有提到。

上下文：
{context}

问题：
{question}
"""
)

llm = ChatOpenAI(model="gpt-4.1-mini", temperature=0)

question = "FastAPI 为什么适合构建 API？"
retrieved_docs = vector_store.similarity_search(question, k=4)
context = "\n\n".join(doc.page_content for doc in retrieved_docs)

chain = prompt | llm
result = chain.invoke({"context": context, "question": question})

print(result.content)
```

## 11. 返回引用来源

基础 RAG 也应该返回引用来源。

```python
sources = []

for doc in retrieved_docs:
    sources.append(
        {
            "source": doc.metadata.get("source"),
            "page": doc.metadata.get("page"),
            "title": doc.metadata.get("title"),
        }
    )

response = {
    "answer": result.content,
    "sources": sources,
}

print(response)
```

接口返回示例：

```json
{
  "answer": "FastAPI 适合构建高性能 API，因为它基于 ASGI，支持异步，并且能自动生成接口文档。",
  "sources": [
    {
      "source": "FastAPI.md",
      "page": 1,
      "title": "FastAPI 入门"
    }
  ]
}
```

## 12. 基础阶段实践任务

完成一个本地知识库问答 Demo：

- 从 `docs/` 目录加载 Markdown 文件
- 使用 `RecursiveCharacterTextSplitter` 分块
- 使用 embedding 生成向量
- 使用 Chroma 保存索引
- 根据用户问题检索 top 4 文档块
- 调用 LLM 生成答案
- 返回答案和引用来源

基础阶段的目标不是优化召回，而是把 RAG 全链路跑通。
