# RAG进阶篇

## 1. 进阶开发的核心变化

基础 RAG 关注系统是否能回答；进阶 RAG 关注回答是否准确、可追溯、可排查。

进阶阶段要解决的问题：

- 用户问题表达不完整，检索不到正确文档。
- top_k 结果相似度高，但不一定真正有用。
- chunk 切分不合理，导致语义断裂。
- 多个知识点分散在不同文档中。
- 答案没有引用来源，无法确认依据。
- 系统不知道错误发生在检索阶段还是生成阶段。

进阶 RAG 的核心是优化检索链路，而不是只换一个更大的模型。

## 2. RAG 项目结构

建议把 RAG 作为独立 service，而不是把所有逻辑写在路由里。

```text
app/
  api/
    rag_router.py
  schemas/
    rag_schema.py
  services/
    rag_service.py
    document_service.py
  retrievers/
    vector_retriever.py
    hybrid_retriever.py
  loaders/
    markdown_loader.py
    pdf_loader.py
  stores/
    vector_store.py
  prompts/
    rag_prompt.py
```

分层职责：

| 层 | 职责 |
| --- | --- |
| router | 接收请求、返回响应 |
| schema | 定义请求和响应结构 |
| service | 编排加载、检索、生成逻辑 |
| retriever | 封装检索策略 |
| store | 封装向量数据库 |
| prompt | 管理提示词模板 |

## 3. 文档元数据设计

metadata 决定了 RAG 系统是否可追溯。

推荐字段：

```python
metadata = {
    "doc_id": "doc_001",
    "source": "FastAPI.md",
    "title": "FastAPI 入门",
    "page": 1,
    "chunk_id": "doc_001_0001",
    "chunk_index": 1,
    "category": "python/Agent",
    "created_at": "2026-06-02",
    "updated_at": "2026-06-02",
}
```

字段说明：

| 字段 | 作用 |
| --- | --- |
| `doc_id` | 标识原始文档 |
| `source` | 文件路径或 URL |
| `title` | 文档标题 |
| `page` | PDF 或文档页码 |
| `chunk_id` | 标识单个文本块 |
| `chunk_index` | 文本块顺序 |
| `category` | 知识库分类 |
| `updated_at` | 支持增量更新 |

没有 metadata 的 RAG 很难进入生产环境。

## 4. 分块策略进阶

不要所有文档都使用同一种 chunk_size。

### 4.1 普通说明文档

适合递归字符切分。

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=120,
    separators=["\n\n", "\n", "。", "，", " ", ""],
)
```

### 4.2 Markdown 文档

Markdown 应优先按标题切分，再按长度切分。

```python
from langchain_text_splitters import MarkdownHeaderTextSplitter

headers_to_split_on = [
    ("#", "h1"),
    ("##", "h2"),
    ("###", "h3"),
]

splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=headers_to_split_on
)

docs = splitter.split_text(markdown_text)
```

这样可以把标题层级保存进 metadata。

### 4.3 FAQ 文档

FAQ 不适合随意按长度切分，最好按问答对切分。

```text
Q: 如何重置密码？
A: 进入设置页面，点击重置密码。
```

一个问答对就是一个 chunk。

### 4.4 API 文档

API 文档适合按接口切分。

一个 chunk 可以包含：

- 接口路径
- 请求方法
- 参数说明
- 响应示例
- 错误码

## 5. Retriever 封装

不要在业务代码里直接调用 `similarity_search`，应该封装 retriever。

```python
from typing import List
from langchain_core.documents import Document

class VectorRetriever:
    def __init__(self, vector_store, k: int = 4):
        self.vector_store = vector_store
        self.k = k

    def retrieve(self, query: str) -> List[Document]:
        return self.vector_store.similarity_search(query, k=self.k)
```

好处：

- 后续可以替换向量库
- 可以统一记录检索日志
- 可以加入过滤、rerank、压缩
- 方便写单元测试

## 6. Metadata Filter

很多业务知识库不是全局检索，而是按用户、租户、分类过滤。

示例：

```python
docs = vector_store.similarity_search(
    query="如何配置 FastAPI 路由？",
    k=5,
    filter={"category": "python/Agent"},
)
```

常见过滤条件：

- 用户 ID
- 租户 ID
- 知识库 ID
- 文档分类
- 创建时间
- 权限范围

权限过滤必须在检索阶段完成，不能只靠 prompt 让模型不要回答。

## 7. 查询改写

用户的问题经常不适合直接检索。

例如：

```text
用户问题：这个怎么部署？
```

如果没有上下文，向量检索很难知道“这个”指什么。

查询改写可以把问题改成更适合检索的形式。

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

rewrite_prompt = ChatPromptTemplate.from_template(
    """
请将用户问题改写为适合知识库检索的查询语句。
要求：
1. 保留用户原始意图。
2. 补全必要上下文。
3. 不要回答问题，只输出改写后的查询。

历史上下文：
{history}

用户问题：
{question}
"""
)

llm = ChatOpenAI(model="gpt-4.1-mini", temperature=0)
rewrite_chain = rewrite_prompt | llm

rewritten = rewrite_chain.invoke(
    {"history": "用户正在学习 FastAPI", "question": "这个怎么部署？"}
)

print(rewritten.content)
```

常见改写策略：

| 策略 | 作用 |
| --- | --- |
| Query Rewrite | 补全问题 |
| Multi Query | 生成多个等价查询 |
| Step-back Query | 抽象成更通用的问题 |
| Query Decomposition | 把复杂问题拆成多个子问题 |

## 8. Multi Query Retriever

Multi Query 的思路是让模型生成多个检索问题，扩大召回范围。

```text
原问题：RAG 回答不准怎么办？

改写为：
1. RAG 检索不到正确文档怎么优化？
2. RAG 如何降低幻觉？
3. RAG chunk 和 top_k 如何调整？
```

适合场景：

- 用户问题很短
- 文档表达方式和用户表达方式差异大
- 需要扩大召回

风险：

- 检索成本上升
- 可能引入更多无关文档
- 需要配合 rerank 或去重

## 9. MMR 检索

MMR 是 Maximal Marginal Relevance，目标是在相关性和多样性之间做平衡。

普通相似度检索可能返回很多相似 chunk，MMR 可以减少重复结果。

```python
retriever = vector_store.as_retriever(
    search_type="mmr",
    search_kwargs={
        "k": 5,
        "fetch_k": 20,
        "lambda_mult": 0.5,
    },
)

docs = retriever.invoke("RAG 如何优化检索质量？")
```

参数说明：

| 参数 | 说明 |
| --- | --- |
| `k` | 最终返回数量 |
| `fetch_k` | 初始候选数量 |
| `lambda_mult` | 相关性和多样性的权重 |

## 10. Prompt 设计进阶

RAG Prompt 需要明确约束模型行为。

```python
from langchain_core.prompts import ChatPromptTemplate

rag_prompt = ChatPromptTemplate.from_messages(
    [
        (
            "system",
            """
你是一个严谨的知识库问答助手。
回答规则：
1. 只能使用给定上下文。
2. 如果上下文没有答案，回答“资料中没有提到”。
3. 不要编造文件名、页码和来源。
4. 回答要简洁，但保留关键步骤。
5. 最后列出使用到的来源编号。
""",
        ),
        (
            "human",
            """
上下文：
{context}

问题：
{question}
""",
        ),
    ]
)
```

上下文可以带编号：

```python
def format_docs(docs):
    formatted = []
    for i, doc in enumerate(docs, start=1):
        source = doc.metadata.get("source", "unknown")
        page = doc.metadata.get("page", "unknown")
        formatted.append(
            f"[{i}] 来源：{source}，页码：{page}\n{doc.page_content}"
        )
    return "\n\n".join(formatted)
```

## 11. FastAPI 接口设计

请求模型：

```python
from pydantic import BaseModel, Field

class RagChatRequest(BaseModel):
    question: str = Field(min_length=1, description="用户问题")
    knowledge_base_id: str = Field(description="知识库 ID")
    top_k: int = Field(default=4, ge=1, le=20)

class SourceItem(BaseModel):
    source: str
    page: int | None = None
    score: float | None = None
    content_preview: str

class RagChatResponse(BaseModel):
    answer: str
    sources: list[SourceItem]
    rewritten_query: str | None = None
    latency_ms: int
```

路由示例：

```python
from fastapi import APIRouter

router = APIRouter(prefix="/rag", tags=["RAG"])

@router.post("/chat", response_model=RagChatResponse)
async def rag_chat(request: RagChatRequest):
    return await rag_service.chat(request)
```
