# RAG高级篇

## 1. 高级 RAG 的主线

高级 RAG 关注生产环境的准确率、稳定性、成本和评估闭环。

基础 RAG 是：

```text
向量检索 top_k -> 拼上下文 -> LLM 回答
```

高级 RAG 是：

```text
查询理解 -> 多路召回 -> 合并去重 -> rerank -> 上下文压缩 -> 生成答案 -> 评估 -> 反馈优化
```

高级阶段的核心能力：

- 混合检索
- rerank 重排
- 上下文压缩
- 增量索引
- 权限隔离
- RAG 评估
- trace 和成本监控
- 失败排查和持续优化

## 2. 混合检索

向量检索擅长语义相似，关键词检索擅长精确匹配。

只用向量检索时，下面这些内容容易召回不稳定：

- 错误码
- API 路径
- 函数名
- 订单号
- 产品型号
- 人名
- 专有名词

混合检索的流程：

```text
用户问题
  -> 向量检索 top_n
  -> BM25 关键词检索 top_n
  -> 合并结果
  -> 去重
  -> rerank
  -> top_k
```

适用场景：

| 场景 | 原因 |
| --- | --- |
| 技术文档问答 | 函数名、路径、错误码需要精确匹配 |
| 企业搜索 | 人名、部门、制度编号需要关键词召回 |
| 客服知识库 | 用户表达多样，需要语义召回 |
| API 文档助手 | 路径和参数名不能只靠语义 |

## 3. BM25 关键词检索

BM25 是经典关键词检索算法，适合补足向量检索的不足。

LangChain 中可以使用 BM25 retriever。

```python
from langchain_community.retrievers import BM25Retriever

bm25_retriever = BM25Retriever.from_documents(chunks)
bm25_retriever.k = 10

docs = bm25_retriever.invoke("FastAPI Depends")
```

BM25 的特点：

- 不需要 embedding
- 对关键词、代码、错误码更敏感
- 不理解语义
- 适合和向量检索组合

## 4. 多路召回合并

可以把多个 retriever 的结果合并。

```python
def merge_documents(*doc_lists):
    seen = set()
    merged = []

    for docs in doc_lists:
        for doc in docs:
            chunk_id = doc.metadata.get("chunk_id") or doc.page_content[:80]
            if chunk_id in seen:
                continue
            seen.add(chunk_id)
            merged.append(doc)

    return merged

vector_docs = vector_retriever.invoke(query)
bm25_docs = bm25_retriever.invoke(query)

candidates = merge_documents(vector_docs, bm25_docs)
```

注意：

- 合并后要去重。
- 不要把候选结果全部塞进 prompt。
- 候选集需要 rerank 或规则排序。

## 5. Rerank 重排

向量库返回的 top_k 不一定是最适合给模型的上下文。

Rerank 的作用是对候选文档重新排序。

常见流程：

```text
召回 30 个候选 chunk -> rerank -> 取前 5 个 chunk -> 进入 prompt
```

rerank 可以使用：

- 专门的 rerank 模型
- cross encoder
- LLM 判断相关性
- 规则分数融合

伪代码：

```python
def rerank(query: str, candidates: list):
    scored = []
    for doc in candidates:
        score = rerank_model.score(query, doc.page_content)
        scored.append((doc, score))
    scored.sort(key=lambda item: item[1], reverse=True)
    return scored
```

工程建议：

- 先召回 20 - 50 个候选。
- rerank 后取 3 - 8 个。
- 保存 rerank 分数。
- 对 rerank 前后的结果都记录日志。

## 6. 上下文压缩

当检索结果很长时，直接拼接会浪费 token，并且容易引入干扰信息。

上下文压缩的目标是只保留和问题相关的内容。

常见方式：

| 方式 | 说明 |
| --- | --- |
| 句子级筛选 | 只保留相关句子 |
| 摘要压缩 | 对长 chunk 先摘要 |
| 结构化提取 | 从表格、列表中抽取关键字段 |
| 去重压缩 | 删除重复描述 |

注意：压缩可能丢失信息，所以必须配合评估。

## 7. 增量索引

生产知识库会持续更新，不能每次都全量重建。

增量索引需要考虑：

- 新增文档
- 修改文档
- 删除文档
- 文档版本
- chunk_id 稳定性
- 向量库删除旧索引

推荐流程：

```text
计算文件 hash -> 判断是否变化 -> 删除旧 chunk -> 重新分块 -> 写入新向量
```

metadata 示例：

```python
metadata = {
    "doc_id": "fastapi_intro",
    "chunk_id": "fastapi_intro_0001",
    "content_hash": "a8f31...",
    "version": 3,
    "updated_at": "2026-06-02",
}
```

## 8. 多租户和权限控制

企业 RAG 必须考虑权限。

错误做法：

```text
先全库检索，再在 prompt 中告诉模型不要回答无权限内容。
```

正确做法：

```text
根据 user_id / tenant_id / role 在检索阶段过滤可访问文档。
```

示例：

```python
docs = vector_store.similarity_search(
    query=query,
    k=5,
    filter={
        "tenant_id": current_user.tenant_id,
        "visibility": "public",
    },
)
```

高权限文档不应该进入模型上下文。

## 9. RAG 评估

RAG 评估要区分检索质量和生成质量。

### 9.1 检索评估

检索评估关注：正确资料有没有被找到。

常见指标：

| 指标 | 含义 |
| --- | --- |
| Hit Rate | top_k 中是否命中正确文档 |
| Recall@k | 正确文档被召回的比例 |
| MRR | 正确文档排名是否靠前 |
| Precision@k | top_k 中相关文档比例 |

### 9.2 生成评估

生成评估关注：答案是否忠于上下文。

常见指标：

| 指标 | 含义 |
| --- | --- |
| Answer Relevance | 答案是否回答了问题 |
| Faithfulness | 答案是否忠于上下文 |
| Context Precision | 检索上下文是否精确 |
| Context Recall | 上下文是否覆盖答案所需信息 |

## 10. 构建评估集

评估集不是越大越好，早期可以先准备 30 - 100 条高质量样本。

每条样本包含：

```json
{
  "question": "FastAPI 中 Depends 的作用是什么？",
  "expected_answer": "Depends 用于声明依赖注入，常用于鉴权、数据库连接、公共参数等。",
  "expected_sources": ["FastAPI.md"],
  "tags": ["FastAPI", "依赖注入"]
}
```

评估集来源：

- 真实用户问题
- 文档标题改写的问题
- FAQ
- 面试题
- 历史错误案例

## 11. Trace 与排障

RAG 错误通常分为三类：

| 类型 | 现象 | 排查方向 |
| --- | --- | --- |
| 检索失败 | 没找到正确文档 | chunk、embedding、top_k、filter |
| 上下文污染 | 找到太多无关内容 | rerank、压缩、metadata |
| 生成失败 | 上下文正确但回答错 | prompt、模型、temperature |

每次请求建议记录：

```json
{
  "request_id": "req_001",
  "question": "FastAPI Depends 有什么用？",
  "rewritten_query": "FastAPI Depends 依赖注入的作用",
  "retrieved_chunks": [
    {
      "chunk_id": "fastapi_001",
      "source": "FastAPI.md",
      "score": 0.83,
      "rerank_score": 0.91
    }
  ],
  "model": "gpt-4.1-mini",
  "prompt_tokens": 1200,
  "completion_tokens": 180,
  "latency_ms": 1560
}
```

## 12. 成本和性能优化

常见优化：

- embedding 批处理
- 缓存相同问题的检索结果
- 缓存高频问题答案
- 控制 top_k 和上下文长度
- 简单问题用小模型
- 复杂问题才走 query rewrite 和 rerank
- 使用流式输出改善体感速度
- 对向量库建立合适索引

缓存策略：

| 缓存对象 | 说明 |
| --- | --- |
| query embedding | 相同问题不用重复向量化 |
| retrieval result | 相同问题不用重复检索 |
| final answer | 高频问题直接返回 |
| document hash | 判断文档是否需要重建索引 |

## 13. 生产级 Checklist

- 文档加载是否稳定
- chunk 是否保留 metadata
- 是否支持增量索引
- 是否有权限过滤
- 是否记录检索结果
- 是否支持混合检索
- 是否有 rerank
- 是否返回引用来源
- 是否有评估集
- 是否统计 token 成本
- 是否有 trace 回放
- 是否有超时、重试和降级
