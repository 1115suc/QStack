# LangGraph高级篇

## 1. 高级 LangGraph 的主线

高级 LangGraph 关注生产级 Agent Harness。

生产级 Agent 不是简单循环调用工具，而是要解决：

- 状态持久化
- 失败恢复
- 人工确认
- 权限控制
- trace 回放
- 多 Agent 协作
- 成本控制
- 安全边界

可以把高级 LangGraph 理解为：

```text
有状态工作流 + 可恢复执行 + 受控工具调用 + 人工介入 + 观测评估
```

## 2. Checkpoint 状态持久化

checkpoint 用于保存图的执行状态。

它适合：

- 多轮对话
- 长任务
- 中断后恢复
- human-in-the-loop
- trace 和 debug

概念流程：

```text
用户请求 -> 执行到某个节点 -> 保存 checkpoint -> 继续或中断 -> 从 checkpoint 恢复
```

示例：

```python
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()

app = graph.compile(checkpointer=checkpointer)

config = {
    "configurable": {
        "thread_id": "user_001_session_001"
    }
}

result = app.invoke(
    {"question": "帮我查询订单 A1001"},
    config=config,
)
```

`thread_id` 用来标识一条会话或一次长期任务。

生产环境中通常不会用 `MemorySaver`，而是使用数据库或 Redis 类持久化方案。

## 3. Human-in-the-loop

人工确认适合高风险节点。

高风险操作：

- 删除数据
- 修改订单
- 发起退款
- 发送邮件
- 执行代码
- 写入数据库
- 调用生产系统 API

设计流程：

```text
Agent 生成计划 -> 命中高风险工具 -> 中断等待确认 -> 用户批准或拒绝 -> 从 checkpoint 继续
```

工程原则：

- 查询类工具可以自动执行。
- 写操作默认 dry-run。
- 真正 commit 前必须确认。
- 确认内容要展示工具名、参数和影响范围。
- 用户拒绝后要能生成解释，而不是静默失败。

## 4. Policy Gate 策略节点

策略节点用于判断某个操作是否允许执行。

```python
from typing import Literal

class ApprovalState(TypedDict):
    action: str
    risk: str
    approved: bool
    answer: str

def policy_node(state: ApprovalState):
    if state["action"] in {"delete_data", "refund", "send_email"}:
        return {"risk": "high"}
    return {"risk": "low"}

def route_by_risk(state: ApprovalState) -> Literal["execute", "need_approval"]:
    if state["risk"] == "high" and not state.get("approved"):
        return "need_approval"
    return "execute"
```

注册条件边：

```python
graph.add_conditional_edges(
    "policy",
    route_by_risk,
    {
        "execute": "execute_tool",
        "need_approval": "approval",
    },
)
```

策略节点适合统一处理权限和风险，而不是散落在每个工具函数里。

## 5. 多 Agent 协作

复杂任务可以拆成多个专业 Agent。

例如研究助手：

| Agent | 职责 |
| --- | --- |
| Planner | 拆解任务 |
| Researcher | 检索资料 |
| Analyst | 分析资料 |
| Writer | 生成报告 |
| Reviewer | 检查事实和格式 |

流程：

```text
用户问题 -> Planner -> Researcher -> Analyst -> Writer -> Reviewer -> END
```

不要为了炫技而多 Agent。只有当任务职责确实不同、上下文需要隔离、步骤需要审计时，多 Agent 才有价值。

## 6. 子图 Subgraph

子图适合把复杂流程模块化。

例如：

```text
主 Agent Graph
  -> RAG Subgraph
  -> Tool Subgraph
  -> Approval Subgraph
```

好处：

- 结构清晰
- 便于复用
- 便于测试
- 避免主图过大

常见子图：

- RAG 子图
- 工具调用子图
- 审批子图
- 报告生成子图
- 数据分析子图

## 7. Trace 与调试

LangGraph 项目要记录每个节点的输入、输出和耗时。

建议记录：

```json
{
  "thread_id": "user_001_session_001",
  "node": "retrieve",
  "input": {
    "question": "FastAPI Depends 有什么用？"
  },
  "output": {
    "documents_count": 4
  },
  "latency_ms": 120,
  "status": "success"
}
```

排查问题时按节点定位：

| 问题 | 排查节点 |
| --- | --- |
| 路由错误 | classify / router |
| 检索错误 | retrieve |
| 工具调错 | tool selector |
| 回答错误 | generate |
| 死循环 | conditional edge / steps |
| 权限问题 | policy |

## 8. 成本和循环控制

Agent 图中必须限制资源使用。

建议限制：

- 最大循环次数
- 最大工具调用次数
- 最大 token 数
- 单个节点超时
- 整体请求超时
- 高成本模型调用次数

状态字段示例：

```python
class RuntimeState(TypedDict):
    question: str
    steps: int
    tool_call_count: int
    total_tokens: int
    total_cost: float
    answer: str
```

路由函数中做预算判断：

```python
def budget_guard(state: RuntimeState):
    if state["steps"] >= 5:
        return "stop"
    if state["total_cost"] > 0.5:
        return "stop"
    return "continue"
```

## 9. 生产级 Graph Checklist

- State 字段是否清晰
- 节点是否职责单一
- 条件路由是否可测试
- 是否限制循环次数
- 是否设置 thread_id
- 是否支持 checkpoint
- 高风险工具是否需要确认
- 是否有 policy gate
- 是否记录每个节点耗时
- 是否记录工具调用参数
- 是否支持失败恢复
- 是否有统一错误处理
- 是否能通过 trace 回放执行过程

## 10. 高级阶段实践任务

实现一个生产级 Agent Graph：

- 支持 `thread_id`
- 支持 checkpoint
- 支持 RAG 分支
- 支持工具调用分支
- 支持高风险操作审批
- 支持最大循环次数限制
- 记录每个节点的输入、输出和耗时
- 用 FastAPI 暴露 `/agent/chat`

高级阶段目标是把 LangGraph 从“流程编排工具”用成“可上线 Agent 工作流引擎”。
