# Agent基础篇

## 1. Agent 解决什么问题

普通 LLM 应用的流程通常是：

```text
用户输入 -> 模型生成文本 -> 返回答案
```

Agent 应用的流程是：

```text
用户输入 -> 模型理解任务 -> 选择工具 -> 执行工具 -> 观察结果 -> 继续推理 -> 返回答案
```

Agent 的核心价值不是“像人一样思考”，而是让大模型可以调用外部工具完成真实任务。

常见任务：

- 查询订单状态
- 检索知识库
- 调用 HTTP API
- 查询数据库
- 读取文件
- 创建工单
- 生成报告
- 执行多步骤业务流程

应用开发工程师学习 Agent，重点不是研究算法，而是学习如何把模型、工具、状态和业务系统安全地组织起来。

## 2. Agent 的核心组成

| 组成 | 作用 |
| --- | --- |
| LLM | 理解用户意图，决定下一步动作 |
| Prompt | 约束 Agent 角色、任务边界和输出方式 |
| Tool | Agent 可以调用的外部能力 |
| Tool Schema | 描述工具名称、参数和用途 |
| Memory | 保存对话历史或长期偏好 |
| State | 保存当前任务执行状态 |
| Executor | 控制 Agent 的执行循环 |
| Trace | 记录模型输入、工具调用和输出 |

基础阶段先理解 LLM + Tool + 执行循环。

## 3. Agent 和 Chain 的区别

Chain 是固定流程。

```text
Prompt -> LLM -> Output Parser
```

Agent 是动态流程。

```text
LLM -> 判断是否调用工具 -> Tool -> LLM -> 继续或结束
```

对比：

| 对比项 | Chain | Agent |
| --- | --- | --- |
| 流程 | 固定 | 动态 |
| 是否调用工具 | 通常固定 | 由模型决定 |
| 适合场景 | 分类、总结、RAG | 多步骤任务、工具调用 |
| 可控性 | 较强 | 需要工程约束 |
| 风险 | 较低 | 工具误调用、循环、权限风险 |

如果一个任务流程非常固定，优先用 Chain 或 LangGraph 固定流程，不要为了使用 Agent 而使用 Agent。

## 4. Tool Calling 是什么

Tool Calling 是模型调用工具的标准能力。

模型不会直接执行 Python 函数，而是生成一个结构化工具调用请求。

```json
{
  "name": "get_order_status",
  "arguments": {
    "order_id": "A1001"
  }
}
```

应用程序收到这个请求后，真正执行工具函数，并把结果返回给模型。

```text
模型请求调用工具 -> 程序执行工具 -> 工具结果返回模型 -> 模型生成最终答案
```

## 5. LangChain 工具定义

最简单的工具：

```python
from langchain_core.tools import tool

@tool
def get_order_status(order_id: str) -> str:
    """查询订单状态。order_id 是订单编号，例如 A1001。"""
    fake_db = {
        "A1001": "已支付，等待发货",
        "A1002": "已发货，运输中",
    }
    return fake_db.get(order_id, "未找到订单")
```

工具设计要点：

- 函数名要表达动作。
- docstring 要写清楚什么时候用。
- 参数名要具体。
- 返回结果要简短、稳定。
- 工具内部要处理异常。

## 6. 工具 Schema

对 Agent 来说，工具 Schema 比工具代码更重要。

坏设计：

```python
@tool
def query(input: str) -> str:
    """查询信息。"""
    ...
```

问题：

- `query` 太泛。
- `input` 不知道应该传什么。
- 模型很难判断什么时候调用。

好设计：

```python
from pydantic import BaseModel, Field
from langchain_core.tools import tool

class OrderStatusInput(BaseModel):
    order_id: str = Field(description="订单编号，例如 A1001")

@tool(args_schema=OrderStatusInput)
def get_order_status(order_id: str) -> str:
    """根据订单编号查询订单状态。"""
    fake_db = {
        "A1001": "已支付，等待发货",
        "A1002": "已发货，运输中",
    }
    return fake_db.get(order_id, "未找到订单")
```

Schema 设计原则：

| 原则 | 说明 |
| --- | --- |
| 字段清晰 | 参数名表达业务含义 |
| 类型明确 | 使用 str、int、Literal、Enum |
| 描述具体 | Field description 要说明格式和示例 |
| 参数收敛 | 不要用一个大字符串承载所有内容 |
| 默认安全 | 高风险参数默认 dry_run |

## 7. 创建最小 Agent

```python
from langchain.agents import create_agent

agent = create_agent(
    model="openai:gpt-4.1-mini",
    tools=[get_order_status],
    system_prompt="你是一个订单客服助手，需要根据工具结果回答用户。",
)

result = agent.invoke(
    {
        "messages": [
            {"role": "user", "content": "帮我查一下订单 A1001 的状态"}
        ]
    }
)

print(result["messages"][-1].content)
```

执行过程大致是：

```text
用户问题 -> LLM 判断需要订单工具 -> 调用 get_order_status -> LLM 根据工具结果回答
```

## 8. 常见工具类型

| 工具类型 | 示例 | 风险 |
| --- | --- | --- |
| 查询工具 | 查询订单、查询天气 | 低 |
| 检索工具 | RAG、搜索文档 | 低 |
| 计算工具 | 价格计算、日期计算 | 低 |
| 写入工具 | 创建工单、发送消息 | 中 |
| 交易工具 | 退款、支付、下单 | 高 |
| 执行工具 | 执行代码、Shell 命令 | 高 |

基础阶段建议只做查询类和检索类工具。

## 9. Agent Prompt

Agent Prompt 需要说明角色、工具边界和回答规则。

```python
system_prompt = """
你是一个严谨的客服 Agent。

规则：
1. 查询订单状态时必须使用 get_order_status 工具。
2. 如果用户没有提供订单号，先追问订单号。
3. 不要编造订单状态。
4. 工具返回未找到订单时，如实告知用户。
5. 回答要简洁。
"""
```

不要只写：

```text
你是一个智能助手。
```

这种 Prompt 对工具使用没有约束。

## 10. 工具调用结果

工具返回结果应简洁，并且适合放回上下文。

不推荐：

```json
{
  "status": 200,
  "data": {
    "order": {
      "...": "几百行原始 JSON"
    }
  }
}
```

推荐：

```json
{
  "order_id": "A1001",
  "status": "已支付，等待发货",
  "updated_at": "2026-06-02"
}
```

原则：

- 返回关键字段。
- 不要返回过长原始数据。
- 错误信息要可理解。
- 敏感字段要脱敏。
