### 面试

#### 1. 接入支付宝微信对接流程？

**回答：**

在我们的项目中，对接支付宝和微信支付主要分为三个阶段：**前期准备**、**环境配置**和**代码开发**。

- **前期准备（账号与资质）：**
    
    - **支付宝：** 我们需要注册支付宝开放平台账号，进行企业实名认证。对于开发测试，我们主要使用**沙箱环境**，在沙箱应用中获取 `APPID`，并配置沙箱账号。
        
    - **微信支付：** 需要在微信支付商户平台注册并获取**商户号（MchID）**，同时注册微信公众号获取 **APPID**，并在商户平台将两者绑定。微信还需要申请 **API 密钥**，如果是退款业务，还必须下载 **API 证书 (p12文件)**。
        
- **环境配置（密钥生成）：**
    
    - **支付宝：** 采用非对称加密（RSA2）。我们需要生成应用公钥和私钥。将**应用公钥**上传到支付宝平台，换取**支付宝公钥**。项目中配置的是“应用私钥”和“支付宝公钥”。
        
    - **微信（V2）：** 在配置文件中设置 AppID、MchID 和 API Key（用于签名）。
        
- **开发接入：**
    
    - 引入对应的 SDK（如支付宝的 `alipay-easysdk` 或微信的 `wxpay-sdk`）。
        
    - 实现核心接口：统一下单（获取二维码链接）、异步回调通知处理、主动查询和退款。

#### 2. 调用三方支付服务如何保证支付安全? （验签）

**回答：**

支付安全主要依靠**非对称加密**和**签名机制**来保证，核心原则是“私钥签名，公钥验签”。

- **请求安全（防篡改）：**
    
    当我们（商户）向支付宝/微信发起请求时，使用我们可以自己持有的**应用私钥**对请求参数进行签名。第三方平台拥有我们的**应用公钥**，收到请求后用公钥验签，确认请求确实来自我们且参数未被篡改。
    
- **响应安全（防伪造）：**
    
    当第三方（如支付宝）给我们返回结果（同步响应或异步回调）时，他们会使用**支付宝私钥**进行签名。我们的系统配置了**支付宝公钥**，收到通知后，必须先用支付宝公钥进行验签。只有验签通过，才说明这是官方发来的真实数据，防止恶意伪造支付成功的回调通知。
    
- **算法选择：** 目前主流且推荐使用的是 **RSA2** 算法。

#### 3. 如何获取支付结果?

**回答：**

获取支付结果通常有两种方式，我们采用“**异步回调为主，主动查询为辅**”的策略：

1. **异步回调（Notify URL）：**
    
    这是最核心的方式。在下单时，我们会传递一个 `notify_url` 给第三方。用户支付成功后，支付宝/微信会向这个地址推送支付结果（POST 请求）。我们需要解析参数、**验签**，通过后更新本地订单状态（如将状态从“付款中”改为“已结算”）并返回 `success` 给第三方。
    
2. **主动查询（Polling）：**
    
    为了防止网络抖动导致回调丢失，我们不能死等回调。系统通常会有一个定时任务或者在前端轮询时，调用第三方的**查询接口**（如支付宝的 `Factory.Payment.Common().query(orderNo)`）。如果查询到状态已支付，同样会更新本地订单状态。

#### 4. 支付如何生成支付二维码?

**回答：**

生成支付二维码分为两个步骤：**获取链接**和**图片渲染**。

1. **获取链接（核心）：**
    
    - 我们调用支付宝的 `AlipayTradePrecreateRequest`（预下单接口）或微信的 `unifiedOrder`（统一下单接口）。
        
    - 接口调用成功后，第三方会返回一个字符串形式的 URL，支付宝叫 `qr_code`，微信叫 `code_url`。
        
2. **生成图片：**
    
    - 拿到这个 URL 字符串后，我们需要将其转换成二维码图片流。
        
    - 项目中我们可以使用 **Hutool** 工具包中的 `QrCodeUtil`，或者 Google 的 **Zxing** 库。
        
    - 代码示例：`QrCodeUtil.generate(url, 300, 300, file)`，将字符串渲染为图片展示给前端或直接返回 Base64 编码。

#### 5. 订单服务 Order，交易服务 Trade，和三方支付是如何关联的?

**回答：**

这涉及到系统的微服务拆分和解耦：

- **订单服务 (Business/Order)：** 负责具体的业务逻辑（如点菜、计算优惠、桌台信息）。它生成的是**业务订单号 (`product_order_no`)**。当用户确认下单后，它不直接请求支付宝，而是将订单信息转换为一个 `TradingVo` 对象，调用交易服务。
    
- **交易服务 (Trade)：** 是一个基础服务模块，专门负责对接各种支付渠道。
    
    - 它接收业务订单，生成**交易单**，并在数据库表 `sl_trading` 中记录，生成全局唯一的**交易单号 (`trading_order_no`)**。
        
    - **关联关系：** 交易表 `sl_trading` 中同时记录了 `product_order_no`（业务订单号）和后续第三方返回的 `transaction_id`（第三方流水号），起到了**桥梁**作用。
        
- **三方支付：** 只与**交易服务**交互。交易服务拿着 `trading_order_no` 作为参数（即 `out_trade_no`）去请求第三方。
    

这种设计的好处是：业务线（外卖、堂食）可以无限扩展，而支付对接逻辑（交易服务）只需要写一套。

#### 6. 请求支付生成二维码、查询支付状态、退款接口有哪些核心参数?

**回答：**

根据代码实现，核心参数如下：

- **请求支付（预下单/统一下单）：**
    
    - `out_trade_no`：商户订单号（即我们交易系统的交易单号，唯一）。
        
    - `total_amount` / `total_fee`：交易金额（支付宝单位元，微信V2单位分）。
        
    - `subject` / `body`：订单标题（如“餐掌柜-餐饮消费”）。
        
    - `notify_url`：回调地址。
        
- **查询支付状态：**
    
    - `out_trade_no`：只需要商户订单号即可查询。
        
- **退款接口：**
    
    - `out_trade_no`：原商户订单号。
        
    - `refund_amount` / `refund_fee`：本次退款金额。
        
    - `out_refund_no`：退款单号（防止重复退款）。

#### 7. 微信接口分为V2、V3版本接口，对接V2版本和支付宝版本有什么区别?

**回答：**

在本项目中，我们对接的是微信 V2 接口（XML格式）和支付宝接口，主要区别在于**交互格式**、**安全机制**和**退款流程**：

1. **交互数据格式：**
    
    - **支付宝**（及微信 V3）：主要使用 **JSON** 格式，或者 Form 表单提交，更加现代化和易读。
        
    - **微信 V2**：请求和响应强制使用 **XML** 格式，我们需要使用 `WXPayUtil.xmlToMap` 进行繁琐的格式转换。
        
2. **签名与加密方式：**
    
    - **支付宝**：强制使用 **RSA2**（非对称加密），需要公钥和私钥。
        
    - **微信 V2**：通常使用 **MD5** 或 **HMAC-SHA256**（对称加密），使用 API Key 进行签名。
        
3. **证书要求（退款）：**
    
    - **支付宝**：普通退款只需要密钥即可，不需要加载证书文件。
        
    - **微信 V2**：涉及资金回滚的接口（如退款），**强制**要求加载 **API 证书 (`apiclient_cert.p12`)**，这在代码中需要读取文件流，配置相对麻烦。
